# Lesson Layer Architecture

*Spec version: 1.0 — 1 April 2026*

## Problem

StudioLoom has pedagogical metadata "layers" on each activity (Bloom's, UDL, grouping, AI rules, scaffolding, time weight). These were added ad-hoc during Dimensions v2. The result:

- **No unified toggle system** — UDL toggle was just added (1 Apr 2026), NM uses a different path (`school_context.use_new_metrics`). Each layer's gating is hand-wired.
- **No consistent pattern** for how layers flow through the four touchpoints: generation → editor → uploads → student pages.
- **No way to add new layers** without touching 8+ files and repeating the same wiring pattern.

We want to add 6-8 more layers (Assessment Checkpoints, Thinking Routines, Scaffolding Fade, ATL Skills, Inquiry Phase, Collaboration Quality, Differentiation Pathways, Self-Regulation). Each must be toggleable, auto-generated by AI, extractable from uploads, and visible in the editor. The architecture must make adding a new layer a **1-2 file change**, not a cross-cutting surgery.

---

## Architecture Overview

```
┌─────────────────────────────────────────────────────────────────┐
│                     LAYER REGISTRY                              │
│  src/lib/layers/registry.ts                                     │
│                                                                 │
│  Single source of truth: every layer defined once as a          │
│  LayerDefinition object. All other systems read from here.      │
└─────────────┬───────────┬──────────────┬───────────────┬────────┘
              │           │              │               │
     ┌────────▼──┐  ┌─────▼─────┐  ┌────▼────┐  ┌──────▼──────┐
     │ GENERATION │  │  EDITOR   │  │ UPLOADS │  │  SUMMARY    │
     │ schemas.ts │  │ Activity  │  │ analyse │  │  BAR +      │
     │ prompts.ts │  │ Block.tsx │  │ .ts     │  │  STUDENT    │
     │            │  │           │  │         │  │  PAGES      │
     └────────────┘  └───────────┘  └─────────┘  └─────────────┘
```

### Core Principle: Define Once, Derive Everywhere

A layer is defined **once** in the registry. The registry exports:
1. **TypeScript types** → `ActivitySection` fields
2. **Schema properties** → `schemas.ts` generation tool schemas
3. **Analysis prompts** → `analyse.ts` upload extraction
4. **Editor UI config** → `ActivityBlock.tsx` tab/badge rendering
5. **Summary visualisation** → `DimensionsSummaryBar.tsx` compact display
6. **Preference key** → `teacher_preferences` toggle name

## 1. Layer Registry

### `src/lib/layers/registry.ts`

```typescript
// ─── Layer Definition ───

export interface LayerFieldDef {
  /** JSON Schema property for AI generation tool schemas */
  schema: Record<string, unknown>;
  /** Field name on ActivitySection (must match TypeScript type) */
  fieldName: string;
}

export interface LayerDefinition {
  /** Unique identifier — used as preference key and CSS class prefix */
  id: string;
  /** Human-readable name shown in Settings + editor tabs */
  label: string;
  /** Short description shown in Settings info box */
  description: string;
  /** Icon SVG path (24x24 viewBox, stroke-based) */
  iconPath: string;
  /** Accent color (hex) for badges, dots, highlights */
  color: string;

  // ─── Preference storage ───
  /** Where the toggle lives in teacher_profiles JSONB.
   *  "teacher_preferences" or "school_context". */
  preferenceStore: "teacher_preferences" | "school_context";
  /** Key within the JSONB object, e.g. "enable_udl" */
  preferenceKey: string;
  /** Default state for new teachers */
  defaultEnabled: boolean;

  // ─── Data model ───
  /** Fields this layer adds to ActivitySection */
  activityFields: LayerFieldDef[];
  /** Fields this layer adds to PageContent (optional, most layers are activity-only) */
  pageFields?: LayerFieldDef[];

  // ─── Editor UI ───
  /** Tab configuration for ActivityBlock (null = no dedicated tab) */
  editorTab?: {
    id: string;
    label: string;
    /** React component path for the tab content (lazy-loaded) */
    component: string;
  };
  /** Badge renderer — returns a React element for the activity card header.
   *  Receives the activity's field values. null = no badge. */
  badgeField?: string;

  // ─── Summary bar ───
  /** Visualisation component for DimensionsSummaryBar.
   *  Receives all sections. null = no summary viz. */
  summaryComponent?: string;

  // ─── Generation ───
  /** Extra prompt guidance injected when generating with this layer enabled.
   *  Appended to buildDesignTeachingContext() or DIMENSIONS_METADATA_INSTRUCTION. */
  generationPromptGuidance?: string;

  // ─── Upload extraction ───
  /** Prompt fragment for extractDimensionsFields() when analysing uploads */
  extractionPromptFragment?: string;
  /** Schema fragment for the extraction AI call */
  extractionSchema?: Record<string, unknown>;

  // ─── Category for Settings page grouping ───
  category: "pedagogy" | "inclusivity" | "assessment" | "design_specific";
  /** Sort order within category on Settings page */
  sortOrder: number;
}
```

```typescript
// src/lib/layers/preferences.ts

export type LayerPreferences = Record<string, boolean>;

/** Fetch teacher's enabled layers from profile API. Client-side. */
export async function fetchLayerPreferences(): Promise<LayerPreferences> {
  const res = await fetch("/api/teacher/profile");
  if (!res.ok) return getDefaultPreferences();
  const data = await res.json();
  const tp = data.teacher_preferences || {};
  const prefs: LayerPreferences = {};
  for (const layer of LAYER_REGISTRY) {
    prefs[layer.id] = tp[layer.preferenceKey] ?? layer.defaultEnabled;
  }
  return prefs;
}

/** Default preferences for new teachers (from registry defaultEnabled). */
export function getDefaultPreferences(): LayerPreferences {
  const prefs: LayerPreferences = {};
  for (const layer of LAYER_REGISTRY) {
    prefs[layer.id] = layer.defaultEnabled;
  }
  return prefs;
}

/** Get enabled layer IDs. */
export function getEnabledLayers(prefs: LayerPreferences): string[] {
  return Object.entries(prefs).filter(([, v]) => v).map(([k]) => k);
}

/** Get LayerDefinitions for enabled layers only. */
export function getEnabledLayerDefs(prefs: LayerPreferences): LayerDefinition[] {
  return LAYER_REGISTRY.filter(l => prefs[l.id]);
}
```

```typescript
import { LAYER_REGISTRY, type LayerPreferences } from "@/lib/layers/registry";

export function buildDimensionsSchema(enabledLayers: string[]): Record<string, unknown> {
```

### Prompt Injection — Dynamic Guidance

```typescript
// src/lib/ai/prompts.ts

import { LAYER_REGISTRY } from "@/lib/layers/registry";

export function buildLayerPromptGuidance(enabledLayers: string[]): string {
  const fragments = LAYER_REGISTRY
    .filter(l => enabledLayers.includes(l.id) && l.generationPromptGuidance)
    .map(l => `**${l.label}:** ${l.generationPromptGuidance}`);

  if (fragments.length === 0) return "";

  return `
## Activity Metadata Layers (REQUIRED)
For every activity section, populate the following metadata fields:

${fragments.join("\n\n")}
`;
}
```

This replaces the current hardcoded `DIMENSIONS_METADATA_INSTRUCTION` string.

### Data Flow

```
Teacher preferences → API route reads prefs → builds schema with enabled layers only →
AI generates activities with those fields → validation strips unknown fields →
stored in content_data JSONB (forward-compatible — extra fields are harmless)
```

**Key design decision:** The AI schema only includes fields for enabled layers. This means the AI doesn't waste tokens on fields the teacher doesn't use. But content_data JSONB stores whatever the AI returns — if a teacher enables a layer later, previously generated content won't have those fields (they'll be undefined), which is fine (editor shows "not set" state).

## 5. Upload Extraction Pipeline

### Current Pattern (analyse.ts)

Pass 2b (`extractDimensionsFields`) fires when Pass 2 omits Bloom/UDL/grouping. It uses a focused schema and prompt to extract these three fields.

### New Pattern: Layer-Aware Extraction

```typescript
// src/lib/knowledge/analyse.ts

import { LAYER_REGISTRY } from "@/lib/layers/registry";

async function extractLayerFields(
  text: string,
  pass1: Pass1Structure,
  enabledLayers: string[]
): Promise<Record<string, unknown>> {
  // Build extraction schema from registry
  const schemaProps: Record<string, unknown> = {};
  const promptFragments: string[] = [];

  for (const layer of LAYER_REGISTRY) {
    if (!enabledLayers.includes(layer.id)) continue;
    if (!layer.extractionSchema) continue;

    Object.assign(schemaProps, layer.extractionSchema);
    if (layer.extractionPromptFragment) {
      promptFragments.push(layer.extractionPromptFragment);
    }
  }

  if (Object.keys(schemaProps).length === 0) return {};

  // Single Haiku call with combined schema
  const result = await callAI<Record<string, unknown>>({
    system: `You are a curriculum analyst. Analyse the uploaded document and extract the requested pedagogical metadata. Be evidence-based.`,
    prompt: `Document type: ${pass1.document_type}\n\n${text.slice(0, 6000)}\n\nExtract:\n${promptFragments.map((f, i) => `${i + 1}. ${f}`).join("\n")}`,
    model: "haiku",
    maxTokens: 3072,
    schema: { type: "object", properties: schemaProps },
  });

  return result;
}
```

**The teacher's enabled layers travel with the extraction call** — if a teacher doesn't have Thinking Routines enabled, we don't waste Haiku tokens extracting them from uploads. If they enable it later, they can re-analyse (existing "Re-analyse" button on knowledge items).

## 6. Student Page Rendering

Layers are primarily teacher-facing (editor + planning). On student pages, only a few layers surface:

| Layer | Student visibility |
|-------|-------------------|
| Bloom | **Hidden** — students don't see bloom levels |
| Grouping | **Visible** — shown as group indicator on activity card |
| UDL | **Hidden** — teacher planning only |
| AI Rules | **Active** — controls Design Assistant behavior (invisible to student) |
| Scaffolding | **Visible** — student sees their tier's sentence starters |
| Time weight | **Hidden** — drives timing, not shown |
| Assessment checkpoint | **Subtle** — "Checkpoint" badge shown so students know this matters |
| Thinking routine | **Visible** — routine name + brief explanation shown as header card |
| ATL skills | **Hidden** — teacher/accreditation only |
| Inquiry phase | **Hidden** — teacher planning only |
| Collaboration quality | **Subtle** — role cards shown when `collaboration_depth >= cooperative` |
| Differentiation | **Active** — student sees their tier's instructions |
| Self-regulation | **Visible** — agency moments shown as choice/reflection prompts |

Student page rendering reads from `content_data` directly (no preference check needed — if the data exists, render it; if not, skip). The teacher decided what to include at generation/editing time.

## 8. Adding a New Layer (Checklist)

When adding Layer N+1, a developer needs to:

1. **Define it in `registry.ts`** — id, label, fields, schema, prompts, UI config
2. **Add TypeScript types** in `types/index.ts` — optional fields on ActivitySection
3. **(Optional) Build a tab component** if the layer needs a dedicated editor tab
4. **(Optional) Build a summary component** if the layer needs a DimensionsSummaryBar segment

That's it. The registry drives everything else automatically:
- Settings toggle → auto-generated from registry
- Editor tabs → filtered from registry
- Editor badges → filtered from registry
- Summary bar → composed from registry
- Generation schema → built from registry
- Generation prompt → assembled from registry
- Upload extraction → composed from registry

**No changes needed to:** `schemas.ts` (reads registry), `prompts.ts` (reads registry), `analyse.ts` (reads registry), `LessonEditor.tsx` (reads registry), `ActivityBlock.tsx` (reads registry), `DimensionsSummaryBar.tsx` (reads registry), `settings/page.tsx` (reads registry).

## 10. Open Questions

1. **Should disabled layers still be included in generation schemas with lower priority?** Current design: disabled layers are excluded entirely from AI schemas. Alternative: include all layers but mark disabled ones as "optional, only if obvious" — means re-enabling a layer retroactively has data for previously generated content. Tradeoff: more tokens per generation call.

2. **Layer presets?** Some teachers might want "IB MYP Full" (all layers on) vs "Quick & Simple" (just Bloom + Grouping). Could add preset buttons to Settings. Low priority.

3. **Per-unit layer overrides?** Currently layers are global (teacher-wide). A teacher might want ATL Skills on their MYP units but not their GCSE units. Could add per-class-unit override in the future. The `class_units` JSONB pattern already supports this (same as NM config).

4. **Summary bar overflow?** With 10+ layers enabled, the summary bar gets crowded. Consider: collapsible "show more" after 5 segments, or a 2-row layout, or a popover detail panel.
