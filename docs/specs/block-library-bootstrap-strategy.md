# Block Library Bootstrap Strategy

**Status:** Decision doc — pre-Dimensions3 dependency
**Created:** 7 April 2026
**Depends on:** `supabase/migrations/060_activity_blocks.sql`, `src/lib/activity-blocks/`, `docs/specs/neutral-criterion-taxonomy.md`
**Consumed by:** Dimensions3 Phase A (Block Library), Phase B (Generation Pipeline)

---

## 1. The Question

Dimensions3 makes Activity Blocks the foundational unit of the generation pipeline. Instead of generating every activity from scratch, the pipeline retrieves proven blocks from a library and only generates new activities for gaps.

But the library starts empty. How do we bootstrap it?

This doc covers 5 decisions:
1. Backfill existing units or start empty?
2. Teaching Moves conversion strategy
3. Initial efficacy scoring
4. Duplicate handling
5. Migration path from knowledge_uploads to content_items

---

## 2. Decision 1: Backfill vs. Start Empty

### Options

**Option A — Backfill existing ~245 units into blocks**
- Extract every activity from every unit's `content_data` JSONB
- Each activity section becomes a block with full Dimensions metadata
- Library starts with ~1,200–1,500 blocks (estimating ~5–6 activities per unit)
- Pros: immediate library richness, the generation pipeline has material from day 1
- Cons: existing activities were generated by the OLD pipeline (no Dimensions metadata, no quality signal, no feedback history). They're untested — no teacher has explicitly validated them as reusable components. Bulk extraction creates noise.

**Option B — Start empty, grow organically**
- Library starts with seed blocks only (Teaching Moves + a small curated set)
- Grows from: (a) new uploads through the Dimensions3 ingestion pipeline, (b) extraction from newly generated units, (c) manual teacher creation
- Pros: every block in the library has a known provenance and quality baseline
- Cons: cold-start problem — early generation runs have no library to draw from, so they generate everything from scratch (which is what we're trying to move away from)

**Option C — Selective backfill (RECOMMENDED)**
- Extract blocks ONLY from units that have evidence of teacher engagement:
  - Units with at least 1 class assigned (actually used, not just generated)
  - Units where `content_data` has been edited (forked, modified — teacher validated the content)
  - Units with student progress data (students actually completed activities)
- Apply conservative efficacy scoring (see Decision 3)
- Supplement with Teaching Moves seed blocks (see Decision 2)

### Decision: Option C — Selective Backfill

**Rationale:** A full backfill pollutes the library with low-confidence blocks (many units were generated and never used). Starting empty punishes early adopters. Selective backfill gives us ~200–400 blocks from genuinely used content, supplemented by ~52 curated Teaching Moves. The generation pipeline has enough to work with from day 1 without drowning in unvalidated material.

### Implementation

**Extraction script:** `scripts/backfill-activity-blocks.mjs`

```
For each unit with (class_units.count > 0 OR content_data was forked):
  1. normalizeContentData(unit.content_data)
  2. For each page → for each section:
     a. Create ActivityBlock with:
        - title: page.title + section index
        - prompt: section.prompt
        - source_type: "generated"
        - source_unit_id: unit.id
        - source_page_id: page.id
        - source_activity_index: section index
        - bloom_level: section.bloom_level ?? infer from prompt text
        - time_weight: section.timeWeight ?? "moderate"
        - grouping: section.grouping ?? "individual"
        - response_type: section.responseType
        - criterion_tags: map section.criterionTags → neutral keys
        - scaffolding: section.scaffolding
        - example_response: section.exampleResponse
        - ai_rules: section.ai_rules
        - udl_checkpoints: section.udl_checkpoints
        - success_look_fors: section.success_look_fors
        - materials_needed: infer from section context
        - efficacy_score: computed (see Decision 3)
        - times_used: 1 (it was used in this unit)
     b. Generate embedding via Voyage AI voyage-3.5
     c. Insert into activity_blocks
  3. Write source_block_id back into the unit's content_data sections
```

**Backfill filters:**
- Skip units with `is_archived = true` AND no student progress
- Skip activity sections with empty or placeholder prompts (< 20 characters)
- Skip units where `content_data` is null or empty (`hasContent()` check)
- Deduplicate: if two sections have > 90% prompt text similarity (cosine on embeddings), keep the one from the more-used unit

**Estimated output:** ~200–400 blocks from ~40–60 actively used units

## 3. Decision 2: Teaching Moves Conversion

Dimensions3 spec says the 52 approved Teaching Moves become seed Activity Blocks (source: Appendix A, reviewed 3 April 2026). They replace the `src/lib/ai/teaching-moves.ts` library as a separate system.

### Conversion mapping

Each Teaching Move maps to a block:

| Teaching Move Field | Block Field |
|--------------------|-------------|
| `name` | `title` |
| `description` | `description` + `prompt` (rewritten as student-facing instruction) |
| `phases` | `design_phase` (use first phase; tag multiple in `tags`) |
| `bloom` | `bloom_level` (take first Bloom level) |
| `grouping` | `grouping` |
| `time` range | `time_weight` (map: 5-10 min = quick, 8-15 min = moderate, 15+ min = extended) |
| `energy` | stored in `tags` as `energy:low/medium/high` |
| `boosts` | stored in `tags` as `boosts:student-agency`, `boosts:cognitive-rigour`, etc. |
| `variations` | stored in `description` as "Variations: ..." |

### Seed block properties

- `source_type`: `"community"` (expert-curated, not teacher-specific)
- `teacher_id`: system teacher account (admin UUID, not a real teacher)
- `is_public`: `true` (available to all teachers)
- `efficacy_score`: `65` (above neutral 50, since expert-curated, but below teacher-validated blocks)
- `times_used`: `0` (no usage history yet)
- Embedding generated via Voyage AI

### Post-conversion cleanup

After seed blocks are inserted and verified:
1. Mark `src/lib/ai/teaching-moves.ts` as deprecated
2. Update any code that imports Teaching Moves to read from the block library instead
3. Delete the file after all references are migrated

## 4. Decision 3: Initial Efficacy Scoring

The `efficacy_score` on `activity_blocks` (0–100, starts at 50) is the primary quality signal for block retrieval ranking.

### Scoring tiers for backfilled blocks

| Source | Base Score | Adjustments |
|--------|-----------|-------------|
| Teaching Moves (seed) | 65 | Expert-curated. No usage data. Above neutral but not proven in-platform. |
| Backfilled from used unit (with student progress) | 55 | Has been delivered to students. +5 for having student data. |
| Backfilled from used unit (assigned but no progress) | 50 | Neutral. Assigned to a class but no evidence of student engagement. |
| Backfilled from forked unit | 55 | Teacher edited the content — implies validation. +5 for teacher engagement. |
| Future: extracted from upload | 50 | Neutral. Teacher uploaded the source material but hasn't used the extracted block yet. |
| Future: manually created | 55 | Teacher deliberately created it — implies intentional quality. |

### Score evolution rules (post-bootstrap)

These rules define how efficacy changes during normal operation:

| Event | Score Change | Rationale |
|-------|-------------|-----------|
| Block used in generated unit (teacher kept it) | +3 | Teacher accepted the AI's block choice |
| Block used and teacher edited it | +1 | Teacher engaged with it but needed changes |
| Block offered but teacher deleted it | -5 | Teacher rejected the block |
| Student completion rate > 80% | +2 | Students engaged with the activity |
| Student avg time within 20% of expected | +1 | Activity timing was accurate |
| Student avg time > 2x expected | -2 | Activity was harder/longer than estimated |
| Teacher re-uses block in another unit | +5 | Strongest signal — teacher deliberately chose it again |
| Block unused for 6+ months | -1/month | Staleness decay (caps at -6 total) |

Score is clamped to [0, 100]. Score changes are applied by the feedback loop (Dimensions3 Stage 6 / System 4).

### Cold-start handling

For the first 3 units a teacher generates, the pipeline should:
1. Retrieve blocks normally (using whatever's available)
2. Generate gap-fill activities for missing slots
3. After teacher review (keeps/edits/deletes), update block scores
4. By unit 4, the library has enough signal to make meaningful retrieval decisions

For brand-new teachers with no uploads and no history, the pipeline falls back entirely to gap generation (Stage 3) — which is exactly how the current pipeline works. No regression.

## 5. Decision 4: Duplicate Handling

### Problem

The same activity can appear across multiple units (teacher reuses a good research activity, or the AI generates similar blocks independently). Without dedup, the library accumulates near-identical blocks that clutter search results.

### Strategy: Soft dedup via embedding similarity

1. **At extraction time:** Before inserting a new block, compute its embedding and search for existing blocks with cosine similarity > 0.92 AND same `teacher_id`.
2. **If duplicate found:**
   - Don't create a new block
   - Increment `times_used` on the existing block
   - Write the existing block's ID as `source_block_id` on the new activity section
   - If the new activity has metadata the existing block lacks (e.g., new `ai_rules` or `scaffolding`), merge it in
3. **If no duplicate found:** Create the new block normally.

### Cross-teacher dedup

For `is_public = true` community blocks, dedup also checks across teachers. Same threshold (0.92 cosine). If a teacher's extracted block is near-identical to a public community block, keep the teacher's private copy (they may customize it later) but link it via `tags: ["similar_to:<community_block_id>"]` for analytics.

### What 0.92 cosine catches

At 0.92 threshold with voyage-3.5 1024-dim embeddings:
- "Sketch 3 different ideas for your solution, explaining the strengths of each" ≈ "Draw 3 solution ideas and explain what makes each one strong" → **DUPLICATE** (same activity, different wording)
- "Sketch 3 ideas" ≈ "Sketch 3 ideas and test them with your partner" → **NOT duplicate** (different scope — peer testing changes the activity)

The threshold can be tuned. Start conservative (0.92), lower if too many duplicates slip through.

## 6. Decision 5: knowledge_uploads → content_items Migration

Dimensions3 §19 defines a new `content_items` table to replace `knowledge_uploads` as the canonical ingestion store. This affects block extraction because blocks currently trace their source via `source_upload_id → knowledge_uploads.id`.

### Migration strategy: Additive, not destructive

1. **Phase A:** Create `content_items` table alongside `knowledge_uploads` (both exist).
2. **Phase A:** New uploads go to `content_items`. Extraction writes `source_content_item_id` on blocks.
3. **Phase B:** Backfill existing `knowledge_uploads` rows into `content_items` with `legacy_upload_id` reference.
4. **Phase B:** Update existing blocks to set `source_content_item_id` where `source_upload_id` is set.
5. **Phase C (post-Dimensions3):** Once all code reads from `content_items`, deprecate and eventually drop `knowledge_uploads`.

### Block FK changes

```sql
-- Add new FK column (nullable, additive)
ALTER TABLE activity_blocks
  ADD COLUMN IF NOT EXISTS source_content_item_id UUID;

-- Index for retrieval
CREATE INDEX IF NOT EXISTS idx_activity_blocks_source_content_item
  ON activity_blocks(source_content_item_id)
  WHERE source_content_item_id IS NOT NULL;
```

The existing `source_upload_id` column stays until Phase C. Code that reads block source should check `source_content_item_id` first, then fall back to `source_upload_id`.

## 8. Success Criteria

The bootstrap is successful when:

1. The library contains 250–500 blocks (200–400 from selective backfill + 52 Teaching Moves)
2. A test generation run for a 6-lesson Design unit retrieves at least 3 blocks from the library (vs. generating everything from scratch)
3. Retrieved blocks have relevance scores > 0.7 for their matched context
4. No duplicate blocks with cosine similarity > 0.92 exist for the same teacher
5. Teaching Moves seed blocks appear in search results with appropriate ranking (not dominating, not buried)
6. The backfill script is idempotent (safe to run multiple times)
7. Existing unit content is unchanged (block extraction is read-only on source units, write-only on activity_blocks)

## 9. Risks and Mitigations

| Risk | Impact | Mitigation |
|------|--------|-----------|
| Backfilled blocks have poor metadata (no bloom, no grouping from old pipeline) | Low retrieval quality | Inference pass: use AI to classify bloom/grouping from prompt text for blocks missing metadata. Batch job, Haiku-tier, ~$0.50 for 400 blocks. |
| Embedding cost for initial backfill | ~$2-5 for 400 blocks at Voyage pricing | Acceptable. One-time cost. |
| Existing units' criterionTags use framework-specific keys | Blocks get wrong neutral tags | Apply the mapping table from `neutral-criterion-taxonomy.md` during extraction. Detect framework from class assignment. |
| Teaching Moves prompts are teacher-facing, not student-facing | Blocks need student-facing `prompt` text | Rewrite each Teaching Move description as a student instruction during conversion. Manual review for the 52 items. |
| Selective backfill misses good activities in unused units | Library smaller than possible | Acceptable. These blocks can be extracted later if the teacher starts using the unit. The extraction function already exists. |
