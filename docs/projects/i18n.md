# Multi-Language Support (i18n)

**Status:** PLANNING (Phase 3-4)
**Estimate:** 10-14 days
**Effort:** High (string extraction + translations)

## What is it

Internationalization (i18n) framework for StudioLoom platform UI. Supports:
- Multiple UI languages (English primary, then Chinese/Spanish/French/Arabic priority 2)
- Right-to-left (RTL) for Arabic
- Complex East Asian typography (CJK: Chinese, Japanese, Korean)
- Locale-specific date/time/number formatting
- Language switcher in user settings
- Teacher-selectable classroom language
- Student view respects teacher's choice + personal setting

Does NOT cover:
- AI-generated content translation (separate problem — content generation happens in teacher's language, students see it as-is)
- Real-time translation UI (no Google Translate-style overlays)
- Teacher content translation (teachers author in one language)

## Why it matters

**International adoption:** StudioLoom targets international schools (ages 11-18, worldwide). Many students are multilingual; many parents don't speak English. Schools in China, SE Asia, Latin America, Middle East can't use a platform without localized UI.

**PIPL/GDPR compliance:** UI in student's language is better data privacy practice (students understand terms/consent).

**Competitive:** Khanmigo, Curipod, ManageBac all support 10+ languages. Lack of i18n is a sales blocker for non-English schools.

**Already referenced:** Platform Architecture doc mentions "English-first, designed for 6-language expansion." Roadmap Phase 4 slots i18n. ELL scaffolding (3-tier) and bilingual classroom toggle are partial foundation.

## What already exists

- **ELL scaffolding:** 3-tier (Introductory/Intermediate/Advanced) sentence starters + guided prompts. These are content-layer, not UI-layer.
- **Bilingual toggle concept:** Mentioned in Platform Architecture but not built (was going to auto-translate scaffolding via AI, decided against).
- **User locale preference:** `teacher_profiles.locale`, `students.preferred_language` columns already exist in schema (migration 050+). Not used yet.
- **Timezone awareness:** Server code respects `teacher_profiles.timezone` for scheduled tasks. Infrastructure is i18n-ready.
- **Framework translation:** Criterion names + rubric text ARE framework-specific, so FrameworkAdapter already does content mapping. i18n applies on top.

## Architecture

**Tech stack:**
- **i18n library:** `next-intl` (Next.js 13+ App Router native, best DX)
- **File format:** JSON namespaced by feature (`common.json`, `unit.json`, `student.json`, `admin.json`, etc.)
- **Locale codes:** `en` (English), `zh` (Simplified Chinese), `es` (Spanish), `fr` (French), `ar` (Arabic)
- **RTL support:** CSS logical properties (`margin-inline`, `padding-inline` instead of left/right) + `dir="rtl"` on html/body via context
- **Pluralization & formatting:** next-intl built-in (handles "1 file" vs "2 files", date formatting per locale)
- **Translation workflow:** Phase 1 strings → Phase 2 professional translation service (Phrase.com or Lingohub) → Phase 3 community crowdsourcing (optional)

**File structure:**
```
/public/locales/
  en/
    common.json          # shared strings (buttons, headers, errors)
    student.json         # student-facing pages
    teacher.json         # teacher dashboard & tools
    unit.json            # unit builder wizard, lesson editor
    admin.json           # admin panel
  zh/
    common.json
    student.json
    teacher.json
    unit.json
    admin.json
  es/
  fr/
  ar/
```

**Common strings pattern:**
```json
{
  "nav": {
    "dashboard": "Dashboard",
    "units": "My Units",
    "teaching": "Teaching",
    "settings": "Settings"
  },
  "buttons": {
    "save": "Save",
    "cancel": "Cancel",
    "delete": "Delete"
  },
  "status": {
    "loading": "Loading...",
    "error": "Something went wrong",
    "success": "Success!"
  }
}
```

**Configuration (next-intl):**
```typescript
// src/i18n.ts
import { getRequestConfig } from 'next-intl/server';

export default getRequestConfig(async ({ locale }) => ({
  messages: (await import(`@/locales/${locale}/common.json`)).default,
}));
```

**React component usage:**
```typescript
// src/components/Header.tsx
import { useTranslations } from 'next-intl';

export default function Header() {
  const t = useTranslations('nav');
  return (
    <header>
      <h1>{t('dashboard')}</h1>
    </header>
  );
}
```

**Routing & locale detection:**
```
/en/student/dashboard       → English student view
/zh/teacher/units           → Chinese teacher view
/ar/student/unit/[id]/page/1 → Arabic student lesson

/student/dashboard          → redirects to /[locale]/student/dashboard
                              (detect from Accept-Language header or saved preference)
```

**Language switcher in settings:**
- Student settings: `/student/settings/language` — radio buttons (English, 中文, Español, Français, العربية)
- Teacher settings: 2-part choice: (1) UI language, (2) Classroom language (what language to present instructions/scaffolding in)
- Selection stored: `students.preferred_language`, `teacher_profiles.ui_locale` + `teacher_profiles.classroom_locale`
- Applied on next page load (or instant if using client context)

**RTL handling (Arabic):**
- `<html dir={isRTL ? 'rtl' : 'ltr'}>` on root
- CSS logical properties: `margin-inline-start` instead of `margin-left`, `padding-inline-end` instead of `padding-right`
- Flex/grid: auto-reverse with flexbox when RTL (test required)
- Icons: some flip horizontally in RTL (arrows, checkmarks) — handle via CSS `transform: scaleX(-1)` with RTL media query
- Text: already auto-mirrors in RTL browsers, no change needed
- **Testing:** Arabic is the only common RTL language in scope; Hebrew deferred

**CJK (Chinese/Japanese/Korean) typography:**
- Font stack: `'Noto Sans SC', 'Noto Sans JP', 'Noto Sans KR', system-ui, sans-serif` (open-source Google fonts)
- Line-height: increased 1.2→1.6 (CJK text is denser)
- Word-breaking: `word-break: break-word` for phones (avoid long lines)
- Punctuation: Chinese uses full-width punctuation (。！？）automatic via font; no UI change needed
- Input validation: accept "3" and "３" (full-width numeral) as equivalent
- No Lighthouse penalty: test CJK text rendering with Core Web Vitals audit

## Connection to existing systems

- **User auth:** `students.preferred_language`, `teacher_profiles.ui_locale`, `teacher_profiles.classroom_locale` columns (already exist in schema, unused)
- **Design Assistant:** Mentor system prompt includes language hint from profile. "Student prefers Spanish" → Haiku knows to respond in Spanish (already works, just needs activation)
- **ELL scaffolding:** Stays as-is (3-tier content, language-independent). i18n applies to UI buttons/labels around it.
- **Framework vocabulary:** FrameworkAdapter already maps neutral → framework-specific. Add language mapping: `FrameworkAdapter + LocaleAdapter = framework + language specific display`.
- **AI content generation:** Teacher sets classroom locale; AI generation uses that language. FUTURE WORK — Phase 5. Current: AI generates in English only.

## Relationship to Dimensions3

Not part of Dimensions3 pipeline. **Phase G (Localization Layer):** Dimensions3 produces content (lessons, blocks, assessments) in teacher's chosen language. i18n handles the platform UI that surrounds the content. Two parallel concerns that intersect at content display. Could be built independently; natural to combine once Dimensions3 content generation is live.

## Build phases

**Phase 1 (3-4 days):** String extraction + next-intl setup + English only (refactor existing hard-coded strings into JSON)
- Audit codebase for hard-coded strings
- Create `common.json`, `student.json`, `teacher.json`, `unit.json`, `admin.json` stubs with English strings
- Refactor all components to use `useTranslations()` hook
- Test English-only flow on all pages
- Set up locale routing in Next.js middleware (detect via header, cookie, query param)

**Phase 2 (5-7 days):** Professional translation to Chinese + Spanish + French + Arabic
- Extract final English strings from Phase 1
- Contract with Phrase.com or Lingohub ($1500-2500 for 5 locales)
- Test translated flows (especially RTL Arabic + CJK typography)
- Set up fallback chain: `zh-Hans` (Simplified) → `zh` → `en` if phrase missing
- Add locale selector UI in student/teacher settings
- Deploy with feature flag (opt-in locale switch, default English)

**Phase 3 (3-5 days):** Community translation portal (optional, Phase 5+)
- Self-hosted: Weblate (free, open-source translation management)
- Teachers/community members can contribute translations
- Voting system for best translations
- Quarterly rollup of community translations into main release

**Phase 4 (future):** AI-generated content localization
- When Dimensions3 content generation is live, add language selection to generation pipeline
- AI generates lesson in teacher's classroom_locale
- Requires passing language through generation prompts (Phase 5+)

## String inventory estimate

- **Common UI:** ~200 strings (buttons, navigation, errors, status messages)
- **Student experience:** ~400 strings (dashboard, lessons, tools, portfolio, gallery)
- **Teacher tools:** ~600 strings (wizard, lesson editor, dashboard, teaching mode, grading)
- **Admin panel:** ~150 strings (reports, settings, analytics)
- **Total:** ~1,350 strings to extract

Translation cost per language: ~$300-500 (at $0.10-0.15 per source word, ~3,500 words total).

## Success metrics

- Platform usable in 5+ languages (functionality parity in all locales)
- CJK typography test passes (no layout breaks, readable font sizes)
- RTL Arabic test passes (layout mirrors correctly, no RTL-specific bugs)
- Language switcher used by 20%+ of teacher base within 6 months
- School adoption in non-English regions increases 30%+ post-i18n
- Translation quality rating (community feedback) >4/5

## Privacy & compliance

- **PIPL (China):** Platform UI in Chinese = better compliance (user understands terms)
- **GDPR:** Offer content in user's language
- **FERPA:** Nothing changes (US schools using English remain English-only)
- **Legal review:** Before launching i18n, have legal review translations for data/privacy terms (non-trivial in Arabic/CJK)

## Notes

- **No machine translation:** Don't use Google Translate API for auto-translation. Professional humans required (especially for legal/data terms).
- **Glossary needed:** Design teaching vocabulary (criterion, scaffolding, rubric, portfolio) must be consistently translated per language. Build glossary per language + train translators.
- **Font loading:** `next/font` with `variable fonts` (Noto Sans supports 100-900 weights, one download). Avoid 4+ separate font files (CJK fonts are large, optimize CDN).
- **Testing:** Use BrowserStack or Sauce Labs to test CJK + RTL rendering on real devices (emulation isn't sufficient for typography edge cases).
- **Community:** Post "Help translate StudioLoom!" on Reddit/Designer Hangout — teachers are often bilingual and motivated.
- **Phased rollout:** Ship Phase 1 (English extracted, routing ready) first. Phase 2 translations hidden behind feature flag. Teachers opt-in to beta. Catch bugs before full launch.

## Risk

- **Translation maintenance burden:** Every UI string change requires re-translating 4 languages. Mitigation: lock UI strings during i18n build (freeze features if possible). Document string-change process for future.
- **CJK typography complexity:** Line breaks, punctuation, font metrics are different. Mitigate with professional QA per language (not automated tests).
- **RTL edge cases:** Arabic layout reveals bugs in dark corners (nested RTL+LTR text, number input alignment, animation directions). Budget 1-2 extra days for RTL QA.
- **AI content still English:** Until Phase 5, AI-generated lessons are English-only even if UI is Chinese. Set expectations: "Lesson content in English, UI in your language."

## Roadmap placement

- **Phase 3 (Stabilization):** Strings extraction + next-intl setup (Phase 1)
- **Phase 4 (Expansion):** Professional translations (Phase 2) + RTL/CJK QA
- **Phase 5 (Internationalization):** Community translation portal + AI content localization
