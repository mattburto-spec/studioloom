## 1. What this phase ships

A single Python worker process that:

1. **Polls** `fabrication_scan_jobs` every 5s via `SELECT ... FOR UPDATE SKIP LOCKED` (per Q2 strategic decision: polling over Realtime).
2. **Downloads** the referenced revision file from the `fabrication-uploads` Supabase Storage bucket via signed URL.
3. **Loads** the active machine profile from `machine_profiles` — rule thresholds are machine-specific.
4. **Runs** all 17 STL rules (R-STL-01 through R-STL-17). Each rule is independent, returns `{id, severity, title, explanation, evidence, fix_hint, version}`.
5. **Renders** a thumbnail (matplotlib isometric + overhang heatmap). Uploads to `fabrication-thumbnails`.
6. **Writes** structured results + updated scan_status into `fabrication_job_revisions`, denormalises `latest_scan_results` + `scan_ruleset_version` onto `fabrication_jobs`.
7. **Dispatches** the `scan_complete` email (using the already-built 1B-2-1 helper with idempotency — so re-scan on the same revision doesn't re-email).
8. **Errors** gracefully on malformed files: writes `scan_status='error' + scan_error` without crashing the worker loop.

**Does NOT** ship in Phase 2A:
- SVG scanning — Phase 2B (same worker, added catalogue).
- Upload UI — Phase 2C.
- Soft-gate results screen — Phase 2D.
- Teacher approval queue — Phase 2E.
- Fabricator pickup — Phase 2F.

## 2. Infrastructure

- **Codebase location**: `fab-scanner/` — sibling to `src/`, `supabase/`, `scripts/` at repo root (per Q3 strategic decision).
- **Deploy target**: Fly.io, app name `preflight-scanner`, region SYD (per Q1).
- **Deploy cadence**: Manual `fly deploy` during 2A; GitHub Action added in Phase 2B (per Q4).
- **Single environment**: production. No staging — for a polling worker, preview infra isn't worth the complexity.
- **Auth**: worker authenticates to Supabase via a dedicated **service-role key**, stored as a Fly secret. The existing service-role key used by Vercel is NOT reused — we want separate rotation surfaces. **Timing: the key is minted at Phase 2A-6** (just before first Fly deploy), not at 2A-1. Sub-phases 2A-1..2A-5 test against a mocked Supabase client so the prod key never touches the laptop during rule-authoring. Forces testable code as a side-effect.

## 3. Sub-phase split (6 instruction blocks)

Each sub-phase gets its own pre-flight → audit → write → test → NC → commit → report cycle. Separate commits. No squashing.

### 2A-1 — Worker scaffold + sandbox (no rules yet)

**Goal:** deployable worker that picks up a job and writes `scan_status='done'` with empty `scan_results`. Proves the polling loop, Supabase auth, signed-URL download, and writeback pipe all work before we add a single rule.

**Files:** full `fab-scanner/` layout above, empty rule modules, end-to-end sandbox test (`python scripts/sandbox.py fixtures/known-good/stl/small-cube-25mm.stl` → prints empty-results JSON).

**Stop trigger:** poll query returns rows locked by another worker. For v1 we run ONE worker instance — that's a Fly config item.

**Commit:** `feat(preflight): Phase 2A-1 worker scaffold + Supabase poll + signed-URL download`

**Est:** ~0.8 day

### 2A-2 — STL rules: geometry integrity (R-STL-01..05)

**Goal:** 5 rules covering watertight, winding, self-intersection, islands, degenerate mesh. All BLOCK severity (except R-STL-03 = WARN, R-STL-04 = WARN). Tests against `known-good/stl/` (expect 0 rules fire) and `known-broken/stl/*` (expect specific rules from sidecar `triggers_rules` fire).

**Commit:** `feat(preflight): Phase 2A-2 STL geometry integrity rules (R-STL-01..05)`

**Est:** ~0.5 day

### 2A-3 — STL rules: machine fit (R-STL-06..08)

**Goal:** 3 rules covering bed size, unit mismatch, print time. Needs machine profile loading from DB. Tests include the 3 authored fixtures (`mount-bracket-oversized`, `chess-pawn-inch-mistake`).

**Commit:** `feat(preflight): Phase 2A-3 STL machine fit rules (R-STL-06..08)`

**Est:** ~0.5 day

### 2A-4 — STL rules: printability (R-STL-09..14)

**Goal:** 6 rules — wall thickness (R-STL-09/10 — ray-cast sampling), overhangs (R-STL-11 — face-normal analysis), feature size (R-STL-12), flat base (R-STL-13), tall+thin stability (R-STL-14). The hardest rules. Thresholds are machine-profile-parameterised.

**Commit:** `feat(preflight): Phase 2A-4 STL printability rules (R-STL-09..14)`

**Est:** ~1.2 day

### 2A-5 — STL informational rules (R-STL-15..17) + thumbnail rendering

**Goal:** 3 FYI rules (print time / filament / triangle count) + matplotlib isometric thumbnail + overhang heatmap. Thumbnails upload to `fabrication-thumbnails` bucket via signed URL (per spec §4 storage path convention).

**Commit:** `feat(preflight): Phase 2A-5 STL informational rules + thumbnail rendering`

**Est:** ~0.8 day

### 2A-6 — Wire end-to-end + Fly deploy + email dispatch + Checkpoint 2.1

**Goal:** Deploy to Fly, test against a real `fabrication_jobs` row created manually, wire `sendFabricationEmail({kind:'scan_complete'})` dispatch post-scan, file a Checkpoint 2.1 report. Requires manual Fly app creation (`fly launch --no-deploy` from `fab-scanner/`), Fly secrets configured, Supabase service-role key minted.

**Stop trigger:** scan hits an R-STL rule in prod that didn't fire locally — investigate fixture/prod divergence.

**Commit:** `feat(preflight): Phase 2A-6 worker deployed to Fly + email dispatch + Checkpoint 2.1`

**Est:** ~0.8 day

## 5. Stop triggers (halt, report, wait for Matt)

- Pre-flight finds `fab-scanner/` directory already exists — halt, investigate greenfield violation.
- Fly.io free trial expired during 2A-6 deploy — halt, ask Matt whether to upgrade billing.
- Supabase service-role key creation requires dashboard access Matt doesn't have — halt.
- Any rule test shows "fixture triggers rule" but sidecar `triggers_rules:` list doesn't include it — halt, investigate: either the sidecar is wrong (edit `write_sidecars.py`) or the rule is over-sensitive.
- A `known-good/stl/*` fixture fires ANY rule — halt. Either the rule is too strict or the fixture belongs in borderline.
- Worker polling holds a lock for >30s on a single job — halt, investigate scan timeout.
- Scan completes but writeback fails (row-level locking, RLS reject) — halt, investigate auth.
- Email dispatch fails at Checkpoint 2.1 — halt, verify Resend + `NEXT_PUBLIC_SITE_URL` propagation.

## 8. Ruleset versioning (new discipline)

Per spec §6.4, every rule carries a `version` string and scan results record `scan_ruleset_version`. When we tune thresholds, we bump the version rather than silently changing behaviour — teachers will ask "why did my file pass last week and fail this week?" and we need an answer.

Initial version: `"stl-v1.0.0"`. Phase 2B adds `"svg-v1.0.0"`. Combined: `"stl-v1.0.0+svg-v1.0.0"`.

Version bump policy:
- **PATCH** (1.0.0 → 1.0.1): rule explanation text tweak, no behaviour change.
- **MINOR** (1.0.0 → 1.1.0): rule threshold loosened (fewer files now fail) OR new rule added.
- **MAJOR** (1.0.0 → 2.0.0): rule threshold tightened OR rule removed.

Bumped in `fab-scanner/src/schemas/ruleset_version.py`. History committed in comments.

## 9. Lessons to re-read (applies to every sub-phase)

- **Lesson #11** — Cache-Control: private on cookie responses. Worker doesn't set cookies, but our email dispatch call may go through a Next.js API that must still set private if it touches session data.
- **Lesson #29** — dual-visibility RLS UNION pattern. Worker writes with service role, so RLS is bypassed — but we should still log which rows we wrote for audit.
- **Lesson #38** — verify expected values, not existence. Rule tests must assert which rules fire, not just "results non-empty."
- **Lesson #41** — NC revert: Edit tool for uncommitted files, git checkout for committed. Applies to every rule-test NC.
- **Lesson #42** — JSONB merge, not replace. `notifications_sent` update on email dispatch must preserve existing keys.
- **Lesson #43** — assumptions block before code. Every sub-phase pre-flight writes one.
- **Lesson #44** — no speculative abstractions. If a rule is trivial (a single `mesh.is_watertight` check), inline it — don't prematurely extract a `RuleEngine` class.
- **Lesson #45** — surgical changes. Sub-phase 2A-N touches only the files listed for that sub-phase.
- **Lesson #51** — Supabase RLS edge cases. Worker auth uses service role — bypasses RLS — but we should still verify writeback row-level constraints.
