## Pre-apply diagnostic — what we found

Pre-fix counts captured against prod with a real student JWT:

| Table | Student JWT can read | Service role sees (ground truth) | Notes |
|---|---|---|---|
| `student_tool_sessions` | 0 | 0 | empty in prod |
| `open_studio_profiles` | 0 | 0 | empty |
| `discovery_sessions` | 2 | 2 | only the test student's own |
| `gallery_submissions` | 0 | 0 | empty |
| `gallery_reviews` | 0 | 0 | empty |
| `own_time_approvals` | **HTTP 404** | **HTTP 404** | **table does not exist in prod** (mig 028 never applied) |
| `own_time_projects` | HTTP 404 | HTTP 404 | same |
| `own_time_sessions` | HTTP 404 | HTTP 404 | same |
| `open_studio_status` | 0 | 0 | empty |
| `open_studio_sessions` | 0 | 0 | empty |
| `class_units` | 2 | 2 | only the test student's enrollments |

**Conclusion:** the IDOR is structurally real (wide-open RLS policies with no role gate) but currently **unexploitable** in prod because the database is sparse — no rows belong to anyone other than the test student. The migration is **defense-in-depth before pilot data arrives**, not an emergency fix.

Two things this surfaced as side-effects:
1. **`own_time_*` tables don't exist in prod** — mig 028 was never applied. Tracked as `FU-PROD-MIGRATION-BACKLOG-AUDIT` (P1). The S1 migration handles this with `to_regclass()` guards so the apply succeeds whether the tables exist or not.
2. **`discovery_sessions` had a SECOND broken policy** beyond the wide-open one — the strict `discovery_sessions_student_select` (mig 047:48-50) compares `students.id::text = JWT.sub` which never matches post-Phase-1.1a (sub is `auth.users.id`, students.id is a different UUID). S1 drops both + replaces with the canonical chain.

If your shell session was reset, re-export:

export PROJECT_URL="https://cxxbfmnbwihuskaaltlk.supabase.co"
export ANON_KEY="<your anon key — Project Settings → API>"
export SERVICE_KEY="<your service_role key — Project Settings → API, Reveal>"

### Step 1 — Apply the migration

Supabase Dashboard → SQL Editor → paste the full contents of:
```
supabase/migrations/20260509034943_rls_hardening_external_review.sql
```
Click **Run**. Wrapped in `BEGIN; ... COMMIT;` — all-or-nothing.

Expected result: success, no errors. The own_time_* sections will silently skip (no-op) because `to_regclass()` returns NULL.

### Step 2 — Post-apply count check (no-regression smoke)

```bash
for T in student_tool_sessions open_studio_profiles discovery_sessions \
         gallery_submissions gallery_reviews \
         open_studio_status open_studio_sessions \
         class_units; do
  STUDENT_RESP=$(curl -s "$PROJECT_URL/rest/v1/$T?select=*" \
    -H "apikey: $ANON_KEY" \
    -H "Authorization: Bearer $STUDENT_JWT" \
    -H "Prefer: count=exact" \
    -H "Range: 0-0" -I 2>/dev/null)
  STUDENT_COUNT=$(echo "$STUDENT_RESP" | grep -i content-range | awk -F'/' '{print $2}' | tr -d '\r\n')
  printf "%-25s student=%s rows\n" "$T:" "${STUDENT_COUNT:-?}"
done
```

**Expected post-migration counts** (should match the pre-migration column-by-column because the test student's own data is still readable; nothing gets newly hidden because there was no foreign data to begin with):

```
student_tool_sessions:    student=0 rows
open_studio_profiles:     student=0 rows
discovery_sessions:       student=2 rows
gallery_submissions:      student=0 rows
gallery_reviews:          student=0 rows
open_studio_status:       student=0 rows
open_studio_sessions:     student=0 rows
class_units:              student=2 rows
```

If `discovery_sessions` or `class_units` drops to 0, **the canonical chain isn't matching** — STOP and investigate (likely auth.uid() vs JWT mismatch). If counts go UP, something is very wrong.

### Step 3 — Live-app smoke

Confirm the app still works end-to-end. The migration should NOT break anything because all routes use service-role admin client which bypasses RLS. Quick check:

1. **Student dashboard** — `/dashboard` as the test student. Class tiles render with their assigned units. (`class_units` via the SSR-client student-units chain — most likely place to surface a regression.)
2. **Toolkit tool save** — open SCAMPER (or any toolkit tool), enter something, save. Reload. State persists. (`student_tool_sessions`.)
3. **Discovery resume** — visit `/discovery`. Existing 2 sessions still load. (`discovery_sessions`.)

If any of these break: paste the browser console error + Network tab response. Down-migration restores pre-S1 state exactly.

### Step 4 — Sign off

Post in chat: "S1.1 ok — apply succeeded, smoke clean". I push the migration commit to `origin/main`, then proceed to S2.

If the apply errored (PostgreSQL syntax error, unexpected dependency), paste the error + I'll diagnose.

## Why the smoke isn't dramatic

In an ordinary pentest narrative, this section would show "before: student JWT reads 10,000 rows across all schools; after: 5 rows of own data". Your numbers don't show that because you're pre-pilot. The win is the policy, not a body count: when the first real cohort joins, their data is private from day one because the lockdown was in place before they showed up.

`scan-rls-coverage.py` will continue to report "clean" both before and after this migration — it counts policies, not policy quality. The known limitation is filed as a follow-up: `FU-SEC-RLS-WIDE-OPEN-SCANNER` (P3) — extend the scanner to flag any `FOR SELECT USING (true)` policy without `TO service_role`.
