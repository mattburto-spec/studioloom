# Key Decisions Log

> Extracted from CLAUDE.md on 7 Apr 2026. This is the canonical record of all architectural and design decisions. CLAUDE.md now points here instead of embedding this content inline.

- **Microsoft OAuth as multi-tenant from day one (30 Apr 2026 PM, Phase 2.1)** — Azure AD app registered as "Multiple Entra ID tenants" / "Allow all tenants" rather than single-tenant scoped to NIS. Single-tenant would have locked Microsoft sign-in to NIS accounts only; multi-tenant lets any Microsoft 365 school onboard via the same app. Cost: zero (Azure registration is free regardless). Trade-off accepted: slightly broader trust surface (any tenant can attempt sign-in) but combined with the per-school `allowed_auth_modes` allowlist (Phase 2.3) we can still gate which auth modes specific schools accept. Supabase tenant URL set to `https://login.microsoftonline.com/common` to match.

- **First-login OAuth teacher provisioning lives in callback route, not Postgres trigger (30 Apr 2026 PM)** — Phase 2.1's callback creates a `teachers` row + sets `app_metadata.user_type='teacher'` when an OAuth user lands without an existing teachers row. A Postgres `BEFORE INSERT` trigger on auth.users would seem cleaner but can't work: Supabase sets `app_metadata` AFTER the row commits (so the trigger can't read provider info to decide actor type), and the trigger can't call `admin.auth.admin.updateUserById()` to set claims. Callback route has both pieces of context (the OAuth user object + admin client) and is testable. Mirrors the lazy-provision pattern from Phase 1.2's classcode-login route.

- **OAuth callback idempotently sets `app_metadata.user_type` on EVERY login, not just first-login (30 Apr 2026 PM)** — Initially Phase 2.1's callback only set the user_type claim during first-login provisioning. But Phase 0's backfill never set `user_type` for teachers (only for students), so existing teachers signing in via OAuth had `user_type: null` in their JWT → Phase 1.3 polymorphic dispatch returned null → routes using requireTeacherSession would 401. Fix: callback now checks `user.app_metadata.user_type` on every OAuth login and calls `admin.updateUserById()` if it's not 'teacher'. Idempotent — the no-op case is fast (one Supabase round-trip per login). Matches the spirit of "never trust the JWT to be fresh; backfill on every callback."

- **Hero empty state when no unit is assigned (30 Apr 2026 PM)** — Teacher dashboard's `NowHero` renders `unitTitle` at 100-108px font size; when `unitTitle` was the fallback `"—"` the giant em-dash + period read as colored placeholder bars (looked like a render bug). Decision: when `vm.unitId` is null, replace the giant title with a meaningful empty state ("No unit assigned. / Pick a unit to teach this class — the hero will fill in.") at smaller typography. Plus: `resolveCurrentPeriod()` now falls back to `cls.units[0]` when the schedule entry's `unitId` is null but the class has class_units assigned (covers the case where the timetable doesn't store unit-period mapping but the class has a unit assigned). Closes FU-DASHBOARD-HERO-NULL-UNIT-TITLE.

- **Split FOR ALL RLS policies into per-cmd when SECURITY DEFINER + INSERT collide (30 Apr 2026 PM)** — CS-2's first hotfix replaced `Teachers manage students` (FOR ALL) with one using `is_teacher_of_student(id)` for both USING and WITH CHECK. The function does `SELECT FROM students WHERE id = student_uuid` — works for SELECT/UPDATE/DELETE (existing row in scope) but breaks INSERT (new row not yet in table when WITH CHECK fires; function returns false; INSERT rejected). Surfaced during UI-INSERT smoke. Fix: split into 4 per-cmd policies. SELECT/UPDATE/DELETE use the SECURITY DEFINER helper as before (existing-row context). INSERT uses direct column comparison `author_teacher_id = auth.uid()` — no function call, recursion-safe by construction (no subquery into RLS-protected tables). General rule: when a SECURITY DEFINER helper function does a table lookup, it can ONLY be used in policies where the row already exists. INSERT WITH CHECK clauses must use direct column expressions or be split into a separate policy. Migration: 20260430053105.

- **POST /api/teacher/students centralizes student creation atomically (30 Apr 2026 PM)** — Closes FU-AV2-UI-STUDENT-INSERT-REFACTOR. Replaced 5 client-side `supabase.from("students").insert()` call sites + 2 helpers with one server-side route that does auth + class-ownership check + INSERT + provisionStudentAuthUserOrThrow + optional class_students enrollment, atomic-ish (rolls back student INSERT on auth failure). Architectural impact: every UI-created student now has auth.users provisioned at create time, not on first login. Closes the NULL user_id security window that was only mitigated by Phase 1.2's lazy-provision fallback. Returns 409 + existingStudentId on duplicate username so the createAndEnroll helper can preserve "exists → enroll" behavior. Bulk imports loop + await per-row instead of batching — round-trip cost acceptable for ~10-30 student typical roster size.

- **Drop legacy `students.class_id` from `/api/student/units` recency-sort (30 Apr 2026 PM)** — Closes FU-AV2-UNITS-ROUTE-CLASS-DISPLAY. The route's class-picker logic was including `students.class_id` (legacy column) as a fallback to the enrollment set. When a unit appeared in both an active class and an archived legacy class, the route picked the archived one for display. Three changes: (1) drop the legacy fallback (Phase 0 backfilled class_students from class_id; junction is canonical), (2) filter classes to non-archived BEFORE the class_units lookup, (3) order enrollments by `enrolled_at DESC` and pick the most-recent match per unit. Behavior change: archived-class-only units stop appearing — correct since students shouldn't work on units from classes they've been removed from.

- **`student_badges.student_id` migrated TEXT → UUID + FK (30 Apr 2026 PM)** — Closes FU-AV2-STUDENT-BADGES-COLUMN-TYPE. Migration 035 created the column as TEXT NOT NULL "nanoid from student_sessions" — meant to hold legacy custom-token nanoids. In practice production always stored text-formatted UUIDs (the comment was speculative). All 3 policies used `::text` casts on both sides. CS-1's rewrite carried the cast forward as a workaround. This migration cleans up: ALTER COLUMN to UUID, ADD FK to students(id) ON DELETE CASCADE, DROP+CREATE all 3 policies without casts. Pre-flight verified 4 rows total, all UUID-shaped, zero orphans. Code callers unchanged because postgres-js + supabase-js auto-coerce string UUIDs.

- **SECURITY DEFINER pattern is the only sustainable shape for cross-table RLS chains (30 Apr 2026, Lesson #64)** — Phase 1.5/1.5b/CS-1 wrote ~14 student-side RLS policies using "canonical chain" subqueries like `target_id IN (SELECT some_col FROM other_table WHERE ...)`. These pass shape tests, work under admin client, and silently recurse the moment SSR client engages. CS-2 hit two cycles in one session; both fixed via SECURITY DEFINER helper functions (`is_teacher_of_student`, `is_teacher_of_class`). New rule: any RLS policy that subqueries another RLS-protected table must use a SECURITY DEFINER wrapper. Direct column comparisons (`user_id = auth.uid()`) stay inline; cross-table joins always wrap. Operational corollary: every future Access-Model-v2 phase that ships RLS policies must include at least one SSR-client smoke test in the same phase as a Checkpoint criterion — not deferred to a follow-up. Otherwise we accumulate latent recursion bombs that fire one at a time in production.

- **Backwards-compatible additive helper refactor for client-switch (30 Apr 2026)** — When CS-2 needed to switch `me/support-settings` + `me/unit-context` to SSR client, the helpers (`resolveStudentSettings`, `resolveStudentClassId`) used `createAdminClient()` internally. Two refactor options: (A) make `supabase` a required parameter — touches all 7 callers including teacher routes + word-lookup; (B) make it optional with `createAdminClient()` as default — only the 2 CS-2 routes opt-in. Picked B. Same shape as Phase 1.4a's dual-mode wrapper — additive change, no churn for unaffected callers. CS-3/CS-N routes opt in as they migrate. Comprehensive "supabase always required" refactor stays a future hygiene step when most/all student routes use SSR client.

- **Frontend login swap is technically Phase 6 cutover landing early (30 Apr 2026)** — Phase 1.2 shipped `/api/auth/student-classcode-login` (sb-* cookies) but the frontend login page (`(auth)/login/page.tsx`) was never switched from legacy `/api/auth/student-login` (questerra_student_session cookie). Phase 1.4b's `requireStudentSession` migration of 6 routes silently regressed all 6 for browser-logged-in students. Caught during CS-2 smoke. The fix — flipping one fetch URL — is technically Phase 6 cutover scope (frontend migrates to new endpoint), but landed early because it's the prerequisite for any Phase 1.4b route to work for real users. Existing legacy-cookie holders keep working via Phase 1.4a's dual-mode wrapper until those cookies expire. Phase 6 still owns the deletion of the legacy `/api/auth/student-login` route + `student_sessions` table.

- **`student_badges.student_id` column-type drift accepted with `::text` cast workaround (30 Apr 2026)** — CS-1's `student_badges_rewrite` migration hit a `text = uuid` operator error: the column is TEXT (created by migration 035 with comment "nanoid from student_sessions") not UUID. The teacher policy works around this with `::text` casts on both sides; we mirrored the cast in the new student policy. Filed FU-AV2-STUDENT-BADGES-COLUMN-TYPE P3 for proper cleanup (ALTER COLUMN to UUID + add FK to students(id) + drop both casts). Hygiene work, not blocking. The new policy works correctly because production rows hold text-formatted UUIDs.

- **Phase 1 closed under Option A — RLS pre-positioned not load-bearing (30 Apr 2026)** — Mid-Phase-1 it became clear that activating Phase 1.5 + 1.5b student-side policies in real route traffic requires (1) authoring student-side policies on supporting tables (`classes`, `units`, `class_units`, etc.) that don't exist yet, and (2) route-by-route switch from `createAdminClient()` to RLS-respecting SSR client with smoke testing per surface. Two options: expand Phase 1 to absorb that work, or close Phase 1 with policies pre-positioned and defer the switch to a P2 follow-up. Picked Option A — defer. Rationale: the new Supabase Auth path is verified end-to-end in prod-preview; existing app-level filtering remains the active line of defense; the dual-mode wrapper means all 63 routes auto-upgrade transparently; client-switch gets cleaner, focused testing as its own deliverable. Filed as FU-AV2-PHASE-14-CLIENT-SWITCH (P2). Trade-off: Phase 1.5 + 1.5b policies are documentation + backstop, not enforcement, until the follow-up ships. Acceptable for pilot — every student route still has explicit `studentId IN (...)` filters in code.
- **Phase 1.4a dual-mode wrapper as zero-risk auto-upgrade (29 Apr 2026)** — Rather than migrate all 63 student routes individually before Phase 1 could close, rewrote the legacy `requireStudentAuth` shim to try `getStudentSession()` first and fall back to the legacy `student_sessions` table lookup if that fails. Net result: every student route auto-upgrades to the new Supabase Auth path with zero route-file changes. Routes that explicitly migrate to `requireStudentSession` (Phase 1.4b) gain access to `session.userId` + `session.schoolId`, but the underlying behaviour is identical for both paths. Removes the "big bang" risk of migrating 63 routes in one phase. Removed at Phase 6 cutover when the legacy cookie path is dropped entirely.
- **Synthetic student email format `<student-id>@students.studioloom.local` (29 Apr 2026)** — Phase 1.2 student login route needs an email to mint Supabase sessions via `generateLink({type:'magiclink'})`. Real student emails aren't collected (under-13 minors, classcode+name flow). Synthetic format uses the student UUID as local-part and `.local` TLD (RFC-reserved, never deliverable). Documented in `provision-student-auth-user.ts` constants. Trade-off: future Phase 2 OAuth flows that try to merge student rows by email will need to filter `.local` domain explicitly.
- **Vercel preview URLs are deployment-specific — use branch-alias for "latest" (29 Apr 2026, Lesson #63)** — Each `git push` mints a new Vercel preview URL with a hash suffix that pins to that exact deployment. Testing yesterday's commit at today's URL hash returns yesterday's behaviour, not the latest deploy on the branch. Use the branch-alias URL pattern `studioloom-git-<branch-with-dashes>-<scope>.vercel.app` to always hit the head of the branch. Cost ~30 min of phantom-401 debugging before realising the fix was a URL change, not a code change.
- **Bug-report auth disambiguation via client hint, server-verified (29 Apr 2026)** — when two valid auth sources can coexist in the same browser (Supabase Auth teacher session + student token cookie), server-side resolution alone always picks one over the other. Frontend now sends `role_hint`; API tries the matching source first and falls through only if it fails. Hint is verified against the matching session, not trusted blindly. Generalises beyond bug-reports — any future endpoint that accepts both auth types should follow this pattern.
- **Bug-report client_context as loose JSONB, not narrow columns (29 Apr 2026)** — userAgent, viewport, connection, hardware, release SHA, route ctx, runtime events all live in one JSONB blob. Schema is intentionally loose / additive — adding a new captured field requires zero migration. Server caps payload at 32 KB (drops `events` array first if oversized, then gives up gracefully). Trade-off: can't index on individual context fields, but at <200 reports/page that doesn't matter.
- **Bug-report screenshot capture: viewport-bounded JPEG over PNG (29 Apr 2026)** — long lesson pages can render to 20+ MP. PNG of that blows past Vercel's 4.5 MB body limit. `toJpeg` q=0.8 with `pixelRatio` dynamically scaled so longest output dim caps at 1400 px (a 1500×8000 page → ~262×1400 ≈ 200–400 KB). 4 MB hard cap server-side, friendly "page too long" message on overflow rather than letting the API 413. Stored in private `bug-report-screenshots` Storage bucket with service-role-only RLS (matches mig 102 pattern); admin GET batch-mints signed URLs (30 min TTL).
- **Bug-report dedupe fingerprint excludes description text (29 Apr 2026)** — `(category | route_kind | first event kind+message)` not `(category | route_kind | description)` because different students will phrase the same bug differently. The intent is "these reports point at the same underlying bug", not "these report descriptions match". Computed client-side over the 200-row page; no schema work. Server-side fingerprint column deferred until report volume exceeds the page cap.
- **Sentry tied to bug reports as a deep-link, not a duplicate store (29 Apr 2026)** — client calls `Sentry.captureMessage` at submit time with bug-report tags so the report is searchable in Sentry alongside auto-captured errors. Returned `event_id` stored on `bug_reports`, admin UI links out. We do NOT duplicate stack traces / breadcrumbs / replay into our DB — those stay in Sentry where they belong.
- **rAF yield required before blocking sync work (29 Apr 2026)** — React state updates set inside an event handler don't paint before subsequent synchronous work in the same async function. If the work that follows blocks the main thread (like `toJpeg`'s DOM/canvas rendering), explicit `requestAnimationFrame` yields are required. Two rAF ticks (one for layout, one for paint) is the safe pattern. Symptom that surfaced this: capture-shimmer never visible despite 2-3 s capture time.

- **Toolkit redesign: phase pills over collections (29 Mar 2026)** — curated collections (beginner, pipeline, quick, research, evaluate) were eliminated. Phase pills (All + Discover/Define/Ideate/Prototype/Test) are the sole filter row. Rationale: collections added cognitive overhead ("what's in each collection?"), phases are universal and immediately understood by design teachers. Toggle behavior: click phase → filter, click same → back to All. Color-matched glows on active state. Auto-scroll to grid on filter click.
- **Toolkit redesign: structural SVG thumbnails over emoji/photos (29 Mar 2026)** — each tool card shows an inline SVG diagram of the tool's OUTPUT SHAPE (mind map = central node with radiating branches, SWOT = 4 colored quadrants, decision matrix = scored grid with checkmarks, fishbone = spine with angled bones, etc.). Phase-colored (discover=blue, define=purple, ideate=green, prototype=amber, test=pink). Better than emoji (too generic), photos (don't communicate structure), or illustrations (expensive to maintain). 26 custom SVGs + fallback (abbreviated initials) for the rest. Function: `svgThumb(id, phase)`.
- **Toolkit redesign: neon gradient line under tabs (29 Mar 2026)** — toolkit category tabs (Design Thinking + 4 greyed "Soon" tabs) sit on a purple-to-blue gradient line with glow. Active tab masks the line beneath it via `border-bottom: 2px solid bg-color` + `margin-bottom: -1px; z-index: 1`. Anchors floating tabs visually.
- **Toolkit redesign: INTERACTIVE badge (29 Mar 2026)** — tools with dedicated `/toolkit/[slug]` interactive pages get a purple "INTERACTIVE" label in top-right corner. Distinguishes the 27 built tools from 21 catalog-only entries. Non-interactive tools will get TemplateToolResponse (guided worksheet) when built.
- **Education AI patterns: effort-gate before feedback (17 Mar 2026)** — all interactive toolkit tools must assess student response quality client-side BEFORE choosing a feedback strategy. Low effort gets pushed for specifics (no praise), high effort gets acknowledged + challenged deeper. This is the core differentiator vs Khanmigo (which gives same feedback regardless of effort). Documented in `docs/education-ai-patterns.md`.
- **Prompts are always read-only (17 Mar 2026)** — students can never click a prompt to auto-fill their response. They must always type their own ideas. Matt's explicit decision — "the student should always be forced to enter their own ideas."
- **Client-side effort assessment over AI-based (17 Mar 2026)** — effort level is determined by word count + linguistic markers (reasoning words, specificity markers) on the client, not by the AI. This gives instant feedback (no API latency), is deterministic (no parsing failures), and the effort level is passed to the API so it can adapt its tone.
- **Phase-aware AI feedback (17 Mar 2026)** — AI nudge tone MUST match the design thinking phase. Ideation tools (SCAMPER, brainstorm, etc.) use divergent encouragement ("what else? push further! wilder!"). Evaluation tools (Decision Matrix, PMI, etc.) use convergent analysis ("what are the trade-offs? who does this fail for?"). Critical questions during ideation kill creative flow — this was a live bug in SCAMPER's first iteration.
- **No auto-deal of prompts (17 Mar 2026)** — prompts are hidden until the student writes their first idea. Student must think independently before scaffolding is offered. After first idea, prompt section slides in and student can choose to deal cards. This reinforces effort-first philosophy.
- **Input-first hierarchy (17 Mar 2026)** — on SCAMPER working screen, the textarea is positioned immediately after the step header. Prompts/scaffolding sit below as secondary support. Example is collapsible ("See an example" toggle) to reduce text clutter. The primary action (writing) should always be above the fold.
- **Four interaction shapes for toolkit tools (17 Mar 2026)** — not all 42 tools should be interactive the same way. Step Sequence (SCAMPER-like steps), Canvas (spatial/visual regions), Comparison Engine (criteria-based scoring with reasoning), Guided Composition (structured writing with coach). ~14 tools are better as templates. Shared components extracted from SCAMPER speed up all future builds. See `docs/ideas/toolkit-interactive-tools-plan.md`.
- **Per-step AI rules are the key differentiator (18 Mar 2026)** — each tool has step/quadrant/column-specific AI rules injected into the system prompt. Six Hats has `hatRules` + `hatTone` per hat; PMI has `colRules` + `colTone` per column; Five Whys has depth-detection logic; Empathy Map has `quadRules` + `quadTone` per quadrant. This means the AI adapts its entire personality per step, not just per tool. No other edtech product does this.
- **Depth detection for Five Whys (18 Mar 2026)** — the AI nudge system prompt includes the student's previous answers chain and explicitly checks if the new answer is "at the same level" (sideways) or "one layer deeper" (good root cause analysis). This is the most pedagogically sophisticated nudge in the toolkit.
- **Empathy Map Feels quadrant pushes contradictions (18 Mar 2026)** — the AI specifically asks students to identify CONTRADICTORY emotions ("excited AND anxious"), pushing past simple happy/sad to nuanced emotional states. This is based on empathy mapping best practice from IDEO and Stanford d.school.
- **Consistent 3-screen architecture (18 Mar 2026)** — all interactive tools use the same intro → working → summary flow. Intro: challenge input + how-it-works. Working: step nav + input-first + prompts + nudge + ideas list. Summary: all steps + AI synthesis + copy. This consistency means students learn the pattern once and can use any tool.
- **All 3 interaction shapes proven (19 Mar 2026)** — Step Sequence (10 tools), Comparison Engine (Decision Matrix), Guided Composition (How Might We). The architectural patterns are validated and can scale to remaining tools.
- **Student toolkit access: two modes, one component (19 Mar 2026)** — Embedded mode (teacher assigns tool on unit page, saves as response, flows to portfolio) and Standalone mode (student opens from floating launcher or nav, saves independently). Same React component, different persistence. Completed sessions are editable (reopen to add ideas) or versionable (start fresh as v2, v3...). Versions show iteration/growth in portfolio — exactly what MYP wants.
- **Public toolkit stays free/unauthenticated (19 Mar 2026)** — `/toolkit` routes remain the free lead-gen tool for teachers. Student versions require auth and persist data. Same components render in both modes.
- **Silent integrity monitoring, not visible to students (19 Mar 2026)** — MonitoredTextarea captures writing behavior with zero visible indicators. Students see a normal textarea. Metadata stored alongside responses in JSONB. Teachers see the IntegrityReport on the grading dashboard. This is a design decision: monitoring is for academic integrity evidence, not surveillance theatre.
- **6-rule integrity scoring over AI-based (19 Mar 2026)** — Human Confidence Score uses deterministic rules (paste ratio, bulk entry, typing speed, editing rate, focus loss, minimal time) not AI classification. Deterministic = explainable to parents/admin, no API cost, instant. Threshold: ≥70 = likely independent, 40-69 = review recommended, <40 = flagged.
- **Timing validation as post-generation guard, not prompt-only (19 Mar 2026)** — Don't just tell the AI to follow Workshop Model in the prompt; validate the output afterward and auto-repair. 8 rules checked, 6 auto-repaired. Separate from generation so prompt changes don't break validation. Schema now collects `workshopPhases` + `extensions` so the AI has the right output structure.
- **Lazy session creation for toolkit persistence (19 Mar 2026)** — useToolSession creates the session on first `updateState()` call, not on mount. Prevents empty session records when students browse without engaging. Debounced auto-save at 500ms coalesces writes.
- **Floating toolkit launcher on all student pages (19 Mar 2026)** — not just during units. Students sometimes need a tool quickly for something outside the current assignment.
- **Fan menu: vertical stack over radial arc (18 Mar 2026)** — radial arc with SVG connecting lines was ugly and clipped off-screen. Replaced with vertical pill stack (phases upward from FAB) + horizontal tool slide-out. Uses Framer Motion springs for entrances/exits. Simpler, more readable, no positioning bugs.
- **Design guidelines as canonical reference (19 Mar 2026)** — `docs/design-guidelines.md` captures 42 principles across 5 categories. Feeds the project dashboard. Will split into shared + project-specific layers when porting to Makloom.
- **Workshop Model is non-negotiable (19 Mar 2026)** — Every AI-generated lesson MUST follow the 4-phase Workshop Model: Opening (5-10 min) → Mini-Lesson (max 1+age min) → Work Time (≥45% of usable time) → Debrief (5-10 min, structured protocol). This is backed by research from d.school, PBLWorks, ASCD, Cult of Pedagogy, Project Zero. Work Time is one sustained block — never fragment it into small activities.
- **1+age rule for instruction cap (19 Mar 2026)** — Maximum direct instruction minutes = 1 + average student age. Year 7 (age 12) = 13 min. Year 10 (age 15) = 16 min. Simpler and more research-backed than the old `maxHighCognitiveMinutes` lookup table. Formula from PBLWorks and cognitive science research.
- **Always use usable time, never raw period (19 Mar 2026)** — The legacy path in `buildTimingBlock()` used to generate content for the full period length when no `TimingContext` was provided. Now it always constructs a default context (theory lesson, 3-min transition). A 60-min workshop lesson has only 41 usable minutes — the AI must generate for 41, not 60.
- **Extensions are mandatory on every lesson (19 Mar 2026)** — Every AI-generated lesson must include 2-3 extension activities for early finishers, indexed to the current design phase. Investigation → deeper research. Ideation → creative variation. Prototyping → alternative materials. Evaluation → edge-case rigor. Extensions are productive deepening, not busywork.
- **Server-side timing validation with auto-repair (19 Mar 2026)** — Don't just prompt the AI to follow rules; validate the output afterward and fix what it gets wrong. 8 rules checked, 6 auto-repaired (missing phases, over-cap instruction, under-floor work time, missing debrief, time mismatch, missing checkpoints). Separate from generation so prompt changes don't break validation.
- **Teacher timing modification via Phase Timeline Bar (19 Mar 2026)** — Teachers drag phase boundaries to redistribute time. Lock phases to prevent auto-resize. One-click presets for common patterns (Balanced, Hands-On Heavy, Instruction Heavy, Critique Session). Post-lesson feedback feeds the learned profile. Teachers know their class better than the AI does — give them the controls.
- **Open Studio replaces Own Time entirely (20 Mar 2026)** — not an extension or parallel system. Migration 029 drops the old tables. Per-unit scoping (not per-class like Own Time). Teacher-only unlock for MVP (auto-calculated criteria later). Configurable check-in interval (5-30 min). Per-unit reset with optional carry-forward.
- **AI mode switching via DB flag, not conversation analysis (20 Mar 2026)** — `open_studio_status.status === 'unlocked'` determines which system prompt is used. Clean, deterministic switch. The Design Assistant checks the DB at response generation time and picks guided or Open Studio prompt.
- **Drift detection with 3-level escalation (20 Mar 2026)** — gentle nudge → direct question → silent flag to teacher. Auto-revocation after 2 consecutive sessions with silent drift flags. Escalation resets per session. Teachers see drift flags in the session log.
- **5 interaction types for Open Studio AI (20 Mar 2026)** — student_message (studio critic tone), check_in (periodic wellness check), drift_check (escalating redirects), documentation_nudge (push to document process), alignment_check (MYP criterion alignment). Each has distinct system prompt instructions.
- **Check-in rotation pattern (20 Mar 2026)** — regular check-in → documentation nudge every 3rd → alignment check every 5th. Prevents monotony. All run on the configurable timer interval.
- **Teaching Mode: polling over Supabase Realtime (20 Mar 2026)** — 8-second polling to `/api/teacher/teach/live-status` for student progress. Simpler than Realtime subscriptions, good enough for classroom use (teacher sees updates within 8s). Can upgrade to Realtime later if needed.
- **Teach Now on main dashboard (20 Mar 2026)** — one-click teaching launch from the main teacher dashboard. Each active unit shows as a card with progress ring. Links to `/teacher/teach/[unitId]?classId=[classId]` so the right class is pre-selected. Reduces friction from 3 clicks to 1.
- **Projector view as separate window, not iframe (20 Mar 2026)** — teacher opens projector view in a new window/tab and drags to projector screen. Dashboard syncs via `postMessage`. Simpler than building a dual-panel layout. Teacher controls stay private on their laptop.
- **China student auth: anonymous by default (20 Mar 2026)** — class code + student-chosen display name, no real names stored in platform. Keeps data anonymized under PIPL. Teacher maintains any name mapping offline. Real-name support requires China-hosted infrastructure (Alibaba Cloud / Tencent Cloud) and standard contractual clauses — a "when you have revenue" problem.
- **Presentation-ready output is a generation problem, not UI (20 Mar 2026)** — the teaching cockpit can only display what the wizard generates. Extending `workshopPhases` schema so each phase carries displayable assets (hook, vocab, checkpoint questions, protocols) is the real fix. Parked on roadmap Phase 4.
- **Projector classroom tools: picker + timer + groups + noise (20 Mar 2026)** — these are the tools teachers tab away for every lesson. Building them into the projector view keeps teachers inside StudioLoom. Noise meter (mic-based volume indicator) is the "wow" feature.
