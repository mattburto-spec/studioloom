# Key Decisions Log

> Extracted from CLAUDE.md on 7 Apr 2026. This is the canonical record of all architectural and design decisions. CLAUDE.md now points here instead of embedding this content inline.

- **AI provider abstraction with fallback chain** — Anthropic → Groq → Gemini, factory pattern
- **Tool use for structured output** — forces JSON schema compliance via tool_choice rather than parsing free text; note: thinking cannot be combined with tool_choice (Anthropic API limitation)
- **Flexible page architecture** — units store content in JSONB; page types: strand, context, skill, reflection, custom
- **Student auth via tokens not Supabase Auth** — nanoid(48) tokens with 7-day TTL for simpler student onboarding
- **Hybrid search over pure vector** — 70/20/10 weighting (vector/BM25/quality) outperforms vector-only
- **Haiku for student-facing, Sonnet for generation** — cost/speed tradeoff; 300-token cap on mentor prevents over-helping
- **Self-healing validation** — invalid AI output gracefully defaults rather than hard failing
- **3-pass analysis for knowledge ingestion** — Structure (Haiku) → Pedagogy (Sonnet) → Design Teaching (Sonnet)
- **Analytics: Plausible or PostHog over GA4** — COPPA/GDPR concerns with tracking minors (ages 11-16)
- **In-memory rate limiting over DB-backed** — resets on Vercel cold start but simplest approach; DB-backed can come later
- **Design Toolkit is framework-agnostic (17 Mar 2026)** — universal phases (Discover/Define/Ideate/Prototype/Test) not MYP-specific, so any design teacher worldwide can use it. Maps to 8+ curricula. This was a critical pivot from the original MYP-only approach.
- **Toolkit at top-level `/toolkit` not under `/tools/`** — gives it its own identity and dark-themed layout separate from the gray `/tools/` layout used by report writer and marking comments
- **Toolkit uses inline styles for dark theme** — the app's Tailwind theme is light; inline styles ensure the dark toolkit theme doesn't leak into other routes
- **Toolkit redesign: phase pills over collections (29 Mar 2026)** — curated collections (beginner, pipeline, quick, research, evaluate) were eliminated. Phase pills (All + Discover/Define/Ideate/Prototype/Test) are the sole filter row. Rationale: collections added cognitive overhead ("what's in each collection?"), phases are universal and immediately understood by design teachers. Toggle behavior: click phase → filter, click same → back to All. Color-matched glows on active state. Auto-scroll to grid on filter click.
- **Toolkit redesign: structural SVG thumbnails over emoji/photos (29 Mar 2026)** — each tool card shows an inline SVG diagram of the tool's OUTPUT SHAPE (mind map = central node with radiating branches, SWOT = 4 colored quadrants, decision matrix = scored grid with checkmarks, fishbone = spine with angled bones, etc.). Phase-colored (discover=blue, define=purple, ideate=green, prototype=amber, test=pink). Better than emoji (too generic), photos (don't communicate structure), or illustrations (expensive to maintain). 26 custom SVGs + fallback (abbreviated initials) for the rest. Function: `svgThumb(id, phase)`.
- **Toolkit redesign: neon gradient line under tabs (29 Mar 2026)** — toolkit category tabs (Design Thinking + 4 greyed "Soon" tabs) sit on a purple-to-blue gradient line with glow. Active tab masks the line beneath it via `border-bottom: 2px solid bg-color` + `margin-bottom: -1px; z-index: 1`. Anchors floating tabs visually.
- **Toolkit redesign: INTERACTIVE badge (29 Mar 2026)** — tools with dedicated `/toolkit/[slug]` interactive pages get a purple "INTERACTIVE" label in top-right corner. Distinguishes the 27 built tools from 21 catalog-only entries. Non-interactive tools will get TemplateToolResponse (guided worksheet) when built.
- **Education AI patterns: effort-gate before feedback (17 Mar 2026)** — all interactive toolkit tools must assess student response quality client-side BEFORE choosing a feedback strategy. Low effort gets pushed for specifics (no praise), high effort gets acknowledged + challenged deeper. This is the core differentiator vs Khanmigo (which gives same feedback regardless of effort). Documented in `docs/education-ai-patterns.md`.
- **Prompts are always read-only (17 Mar 2026)** — students can never click a prompt to auto-fill their response. They must always type their own ideas. Matt's explicit decision — "the student should always be forced to enter their own ideas."
- **Client-side effort assessment over AI-based (17 Mar 2026)** — effort level is determined by word count + linguistic markers (reasoning words, specificity markers) on the client, not by the AI. This gives instant feedback (no API latency), is deterministic (no parsing failures), and the effort level is passed to the API so it can adapt its tone.
- **Phase-aware AI feedback (17 Mar 2026)** — AI nudge tone MUST match the design thinking phase. Ideation tools (SCAMPER, brainstorm, etc.) use divergent encouragement ("what else? push further! wilder!"). Evaluation tools (Decision Matrix, PMI, etc.) use convergent analysis ("what are the trade-offs? who does this fail for?"). Critical questions during ideation kill creative flow — this was a live bug in SCAMPER's first iteration.
- **No auto-deal of prompts (17 Mar 2026)** — prompts are hidden until the student writes their first idea. Student must think independently before scaffolding is offered. After first idea, prompt section slides in and student can choose to deal cards. This reinforces effort-first philosophy.
- **Input-first hierarchy (17 Mar 2026)** — on SCAMPER working screen, the textarea is positioned immediately after the step header. Prompts/scaffolding sit below as secondary support. Example is collapsible ("See an example" toggle) to reduce text clutter. The primary action (writing) should always be above the fold.
- **Four interaction shapes for toolkit tools (17 Mar 2026)** — not all 42 tools should be interactive the same way. Step Sequence (SCAMPER-like steps), Canvas (spatial/visual regions), Comparison Engine (criteria-based scoring with reasoning), Guided Composition (structured writing with coach). ~14 tools are better as templates. Shared components extracted from SCAMPER speed up all future builds. See `docs/ideas/toolkit-interactive-tools-plan.md`.
- **Per-step AI rules are the key differentiator (18 Mar 2026)** — each tool has step/quadrant/column-specific AI rules injected into the system prompt. Six Hats has `hatRules` + `hatTone` per hat; PMI has `colRules` + `colTone` per column; Five Whys has depth-detection logic; Empathy Map has `quadRules` + `quadTone` per quadrant. This means the AI adapts its entire personality per step, not just per tool. No other edtech product does this.
- **Depth detection for Five Whys (18 Mar 2026)** — the AI nudge system prompt includes the student's previous answers chain and explicitly checks if the new answer is "at the same level" (sideways) or "one layer deeper" (good root cause analysis). This is the most pedagogically sophisticated nudge in the toolkit.
- **Empathy Map Feels quadrant pushes contradictions (18 Mar 2026)** — the AI specifically asks students to identify CONTRADICTORY emotions ("excited AND anxious"), pushing past simple happy/sad to nuanced emotional states. This is based on empathy mapping best practice from IDEO and Stanford d.school.
- **Consistent 3-screen architecture (18 Mar 2026)** — all interactive tools use the same intro → working → summary flow. Intro: challenge input + how-it-works. Working: step nav + input-first + prompts + nudge + ideas list. Summary: all steps + AI synthesis + copy. This consistency means students learn the pattern once and can use any tool.
- **All 3 interaction shapes proven (19 Mar 2026)** — Step Sequence (10 tools), Comparison Engine (Decision Matrix), Guided Composition (How Might We). The architectural patterns are validated and can scale to remaining tools.
- **Student toolkit access: two modes, one component (19 Mar 2026)** — Embedded mode (teacher assigns tool on unit page, saves as response, flows to portfolio) and Standalone mode (student opens from floating launcher or nav, saves independently). Same React component, different persistence. Completed sessions are editable (reopen to add ideas) or versionable (start fresh as v2, v3...). Versions show iteration/growth in portfolio — exactly what MYP wants.
- **Public toolkit stays free/unauthenticated (19 Mar 2026)** — `/toolkit` routes remain the free lead-gen tool for teachers. Student versions require auth and persist data. Same components render in both modes.
- **Silent integrity monitoring, not visible to students (19 Mar 2026)** — MonitoredTextarea captures writing behavior with zero visible indicators. Students see a normal textarea. Metadata stored alongside responses in JSONB. Teachers see the IntegrityReport on the grading dashboard. This is a design decision: monitoring is for academic integrity evidence, not surveillance theatre.
- **6-rule integrity scoring over AI-based (19 Mar 2026)** — Human Confidence Score uses deterministic rules (paste ratio, bulk entry, typing speed, editing rate, focus loss, minimal time) not AI classification. Deterministic = explainable to parents/admin, no API cost, instant. Threshold: ≥70 = likely independent, 40-69 = review recommended, <40 = flagged.
- **Timing validation as post-generation guard, not prompt-only (19 Mar 2026)** — Don't just tell the AI to follow Workshop Model in the prompt; validate the output afterward and auto-repair. 8 rules checked, 6 auto-repaired. Separate from generation so prompt changes don't break validation. Schema now collects `workshopPhases` + `extensions` so the AI has the right output structure.
- **Lazy session creation for toolkit persistence (19 Mar 2026)** — useToolSession creates the session on first `updateState()` call, not on mount. Prevents empty session records when students browse without engaging. Debounced auto-save at 500ms coalesces writes.
- **Floating toolkit launcher on all student pages (19 Mar 2026)** — not just during units. Students sometimes need a tool quickly for something outside the current assignment.
- **Fan menu: vertical stack over radial arc (18 Mar 2026)** — radial arc with SVG connecting lines was ugly and clipped off-screen. Replaced with vertical pill stack (phases upward from FAB) + horizontal tool slide-out. Uses Framer Motion springs for entrances/exits. Simpler, more readable, no positioning bugs.
- **Design guidelines as canonical reference (19 Mar 2026)** — `docs/design-guidelines.md` captures 42 principles across 5 categories. Feeds the project dashboard. Will split into shared + project-specific layers when porting to Makloom.
- **Workshop Model is non-negotiable (19 Mar 2026)** — Every AI-generated lesson MUST follow the 4-phase Workshop Model: Opening (5-10 min) → Mini-Lesson (max 1+age min) → Work Time (≥45% of usable time) → Debrief (5-10 min, structured protocol). This is backed by research from d.school, PBLWorks, ASCD, Cult of Pedagogy, Project Zero. Work Time is one sustained block — never fragment it into small activities.
- **1+age rule for instruction cap (19 Mar 2026)** — Maximum direct instruction minutes = 1 + average student age. Year 7 (age 12) = 13 min. Year 10 (age 15) = 16 min. Simpler and more research-backed than the old `maxHighCognitiveMinutes` lookup table. Formula from PBLWorks and cognitive science research.
- **Always use usable time, never raw period (19 Mar 2026)** — The legacy path in `buildTimingBlock()` used to generate content for the full period length when no `TimingContext` was provided. Now it always constructs a default context (theory lesson, 3-min transition). A 60-min workshop lesson has only 41 usable minutes — the AI must generate for 41, not 60.
- **Extensions are mandatory on every lesson (19 Mar 2026)** — Every AI-generated lesson must include 2-3 extension activities for early finishers, indexed to the current design phase. Investigation → deeper research. Ideation → creative variation. Prototyping → alternative materials. Evaluation → edge-case rigor. Extensions are productive deepening, not busywork.
- **Server-side timing validation with auto-repair (19 Mar 2026)** — Don't just prompt the AI to follow rules; validate the output afterward and fix what it gets wrong. 8 rules checked, 6 auto-repaired (missing phases, over-cap instruction, under-floor work time, missing debrief, time mismatch, missing checkpoints). Separate from generation so prompt changes don't break validation.
- **Teacher timing modification via Phase Timeline Bar (19 Mar 2026)** — Teachers drag phase boundaries to redistribute time. Lock phases to prevent auto-resize. One-click presets for common patterns (Balanced, Hands-On Heavy, Instruction Heavy, Critique Session). Post-lesson feedback feeds the learned profile. Teachers know their class better than the AI does — give them the controls.
- **Open Studio replaces Own Time entirely (20 Mar 2026)** — not an extension or parallel system. Migration 029 drops the old tables. Per-unit scoping (not per-class like Own Time). Teacher-only unlock for MVP (auto-calculated criteria later). Configurable check-in interval (5-30 min). Per-unit reset with optional carry-forward.
- **AI mode switching via DB flag, not conversation analysis (20 Mar 2026)** — `open_studio_status.status === 'unlocked'` determines which system prompt is used. Clean, deterministic switch. The Design Assistant checks the DB at response generation time and picks guided or Open Studio prompt.
- **Drift detection with 3-level escalation (20 Mar 2026)** — gentle nudge → direct question → silent flag to teacher. Auto-revocation after 2 consecutive sessions with silent drift flags. Escalation resets per session. Teachers see drift flags in the session log.
- **5 interaction types for Open Studio AI (20 Mar 2026)** — student_message (studio critic tone), check_in (periodic wellness check), drift_check (escalating redirects), documentation_nudge (push to document process), alignment_check (MYP criterion alignment). Each has distinct system prompt instructions.
- **Check-in rotation pattern (20 Mar 2026)** — regular check-in → documentation nudge every 3rd → alignment check every 5th. Prevents monotony. All run on the configurable timer interval.
- **Teaching Mode: polling over Supabase Realtime (20 Mar 2026)** — 8-second polling to `/api/teacher/teach/live-status` for student progress. Simpler than Realtime subscriptions, good enough for classroom use (teacher sees updates within 8s). Can upgrade to Realtime later if needed.
- **Teach Now on main dashboard (20 Mar 2026)** — one-click teaching launch from the main teacher dashboard. Each active unit shows as a card with progress ring. Links to `/teacher/teach/[unitId]?classId=[classId]` so the right class is pre-selected. Reduces friction from 3 clicks to 1.
- **Projector view as separate window, not iframe (20 Mar 2026)** — teacher opens projector view in a new window/tab and drags to projector screen. Dashboard syncs via `postMessage`. Simpler than building a dual-panel layout. Teacher controls stay private on their laptop.
- **China student auth: anonymous by default (20 Mar 2026)** — class code + student-chosen display name, no real names stored in platform. Keeps data anonymized under PIPL. Teacher maintains any name mapping offline. Real-name support requires China-hosted infrastructure (Alibaba Cloud / Tencent Cloud) and standard contractual clauses — a "when you have revenue" problem.
- **Presentation-ready output is a generation problem, not UI (20 Mar 2026)** — the teaching cockpit can only display what the wizard generates. Extending `workshopPhases` schema so each phase carries displayable assets (hook, vocab, checkpoint questions, protocols) is the real fix. Parked on roadmap Phase 4.
- **Projector classroom tools: picker + timer + groups + noise (20 Mar 2026)** — these are the tools teachers tab away for every lesson. Building them into the projector view keeps teachers inside StudioLoom. Noise meter (mic-based volume indicator) is the "wow" feature. Skip: stopwatch, behaviour tracking (ClassDojo territory), seating chart.
- **Pop art visual identity for NM (20 Mar 2026)** — NM components have a deliberately different visual language from the rest of StudioLoom. Hot pink (#FF2D78), electric yellow (#FFE135), cyan (#00D4FF) with bold black borders, halftone dot patterns, box shadows, Arial Black font. This makes NM feel like a distinct "layer" overlaid on the platform — students instantly recognise NM checkpoints. Inspired by Andy Warhol's pop art.
- **NM starts with Agency in Learning only (20 Mar 2026)** — the Melbourne framework has 7 competencies but only Agency in Learning has fully defined elements + progressions in the constants file. Other competencies return empty element arrays. Expand as Matt gets the full Melbourne kits.
- **NM config stored as JSONB on units table (20 Mar 2026)** — `nm_config` column on `units` table stores `{ enabled, competencies, elements, checkpoints: { [pageId]: { elements } } }`. Per-unit, not per-class. Teacher picks which elements to track and assigns them to checkpoint pages via 3-step wizard.
- **NM page_id is TEXT not UUID (20 Mar 2026)** — discovered that StudioLoom page IDs are nanoid(8) strings, not UUIDs. Migration 032 fixes the column type. This is a recurring pattern — any table referencing page IDs must use TEXT.
- **NM observation accessible from Teaching Mode (20 Mar 2026)** — teacher can observe a student's competency in real-time during class via a button on each student card in the teaching cockpit. This is the intended workflow: watch a student, tap NM, rate, move on. Much faster than navigating to the unit page.
- **Unit-as-template architecture / Canvas Blueprint pattern (20 Mar 2026)** — Units are content templates, not per-class instances. `class_units` junction table stores per-class configuration overrides (NM config, future: timing, Open Studio settings). Inspired by Canvas LMS Blueprint Courses. NM config inheritance: `class_units.nm_config` → `units.nm_config` → `DEFAULT_NM_CONFIG`. This means a teacher can enable NM for one class but not another, using the same unit content.
- **NM config belongs on class-unit settings page, not unit page (20 Mar 2026)** — Matt's explicit feedback: "I don't know what this little NM dropdown does, I don't want to control it at this high level." NM is a per-class pedagogical decision. Unit page shows WHAT classes use this unit. Class-unit page controls HOW they use it. Clean separation of content (unit) vs configuration (class-unit).
- **Assigned Classes section on unit detail page (20 Mar 2026)** — Shows which classes are assigned to a unit with student count, class code, NM badge, and archived state. Each card links to `/teacher/units/[unitId]/class/[classId]` for per-class settings. This makes the unit→class relationship visible and navigable. Previously, the only way to see class assignments was via the dashboard.
- **Auth helper pattern for teacher routes (20 Mar 2026)** — `src/lib/auth/verify-teacher-unit.ts` centralises 3 common auth checks: teacher owns unit, teacher owns class, get NM config with fallback chain. Replaces duplicated auth+query logic across NM routes. Pattern: `createServerClient` for auth only, `createAdminClient` for all DB operations (bypasses RLS).
- **Student route links must include sub-path (20 Mar 2026)** — Student unit route has no index page at `/unit/[unitId]` — only `/narrative` and `/[pageId]` sub-routes exist. Links to `/unit/${unitId}` cause Next.js prefetch 404s. Always use `/unit/${unitId}/narrative` as the fallback link target.
- **Migration 033 is additive-only by design (20 Mar 2026)** — Only adds columns and indexes, never modifies or drops existing columns. This means the migration can be applied safely and code can be reverted independently if issues arise. The new columns default to NULL so existing data is unaffected.
- **Effort-gated reflections over tickbox (20 Mar 2026)** — Old reflections (confidence sliders, checklists, empty textareas) were "tickbox exercises" requiring zero thought. Replaced with unified written-response requiring 12+ meaningful words (excluding filler). Confidence slider only appears AFTER writing threshold met. Sentence starters guide quality ("Start with 'I noticed that...'", "Be specific — name the thing", "Connect this to something you actually did"). Effort tags provide instant visual feedback. This applies the same effort-gating pattern used in toolkit tools to the core lesson flow.
- **Comic strip over chat bubbles for Discovery (20 Mar 2026)** — First attempt at visual DiscoveryFlow was styled chat bubbles with comic-style borders. Matt's feedback (with reference image of real illustrated comic) showed he wanted actual scene panels with illustrations. Second iteration added SVG scene illustrations per step, detailed character drawings (mentor + student), thick panel borders, speech bubble tails, cinematic step transitions. Scott McCloud's "Understanding Comics" principles: panel closure (reader fills gaps between panels), progressive reveal (panels animate in with spring physics), visual storytelling. The comic strip replaces the two-column chat+profile layout entirely.
- **No side profile panel in Discovery (20 Mar 2026)** — The old DiscoveryFlow had a right column showing "Start sharing to build your profile" that was confusing (Matt asked "what is the column on the right?"). Removed entirely. Profile data (strengths, interests, needs) now appears inline as ProfileReveal cards within the comic strip flow when the student transitions between phases. Bottom bar shows project statement + archetype badge. Simpler, more focused, no dead space.
- **Sticky JourneyMap in Open Studio (20 Mar 2026)** — The 4-phase journey map (Discovery → Planning → Working → Sharing) is always visible at top of Open Studio chat via `position: sticky`. Auto-compacts when scrolled (shrinks to dot indicators with current phase label). Frosted glass background with blur. This ensures students always know where they are in the process without scrolling up.
- **Sticky top nav on student lesson pages (20 Mar 2026)** — Dashboard button was buried in the hero gradient as faint white text — invisible to students. Moved to a sticky top bar with frosted glass, always visible. Includes page counter, hamburger menu for mobile sidebar, and prominent Dashboard button. Students can always get back to their dashboard in one click.
- **Discovery redesign: story not interview (21 Mar 2026)** — Research-backed pivot. Current Discovery is an interrogation in costume. New approach: mentor tells stories first (campfire effect), student responses to scenarios reveal their profile indirectly. "What do you actually do when your friend needs help?" reveals strengths without asking "What are your strengths?" Based on adolescent psychology research: direct questioning is the #1 trust killer with teens. Provocation over interrogation. Mentor vulnerability before student vulnerability. Every panel is a moment, not a message.
- **Choose Your Mentor before Discovery (21 Mar 2026)** — 5 selectable mentors (Kit/Sage/River/Spark/Haven) with distinct personalities, voices, visual designs, and teaching approaches. Same 5-chapter flow, same data extraction, different experience. The mentor choice itself is a data point — which personality a student gravitates toward reveals something about them before they've answered a question. Stored in DiscoveryProfile. Feeds into AI behaviour for the rest of Open Studio (the system knows not just WHAT the student said but HOW they chose to be spoken to).
- **Student character not illustrated (21 Mar 2026)** — The student is the camera/reader, not a drawn character. Any illustrated student would feel wrong to half the audience. Student speech bubbles rise from the bottom of panels — their voice entering the scene. Graphic novel convention: protagonist's perspective shown through the camera, not through a drawn character.
- **Environmental panels shared, character panels per-mentor (21 Mar 2026)** — The door, collection wall, window, crossroads corridor, and rooftop are the same for all mentors (shared assets). Only character images (~6 per mentor), dialogue, voice, and AI prompts differ. This keeps the production scope manageable: ~45 total images, not 5× everything.
- **Two-layer audio: narration + dialogue (21 Mar 2026)** — Inner monologue (Kit's private thoughts, Max Payne style) plays automatically as panels scroll into view. Dialogue matches speech bubbles. Narration builds character depth the student can't get from text alone ("They don't see it yet — but I do. That instinct they described? That's wiring."). ElevenLabs TTS for pre-recorded clips (~45 per mentor), real-time generation for AI-response panels.
- **Provocation-based strength discovery (21 Mar 2026)** — "Your best friend is panicking, big project due tomorrow, 2 hours left. What do you actually do?" A maker grabs materials. A researcher opens tabs. A leader draws a plan. The student reveals their archetype without being asked to self-categorise. Same principle for interests: "What annoys you?" surfaces authentic interests better than "What are you passionate about?" Passion is abstract; irritation is concrete.
- **Pace-only student feedback (21 Mar 2026)** — Of all possible post-lesson metrics (pace, difficulty, confidence, comprehension, engagement), pace is the only one worth collecting via self-report. Comprehension is better measured from behavioral data (response quality, time spent, integrity metrics). Difficulty and confidence are too abstract for students to self-assess reliably. Pace is concrete ("was this too fast or too slow?") and directly actionable for the timing model. One tap, done. Data flows into `lesson_feedback.feedback_data.pace` → aggregated as `pace_distribution` → feeds timing model for future lesson generation.
- **Student feedback uses student auth, not teacher auth (21 Mar 2026)** — StudentFeedbackPulse posts to `/api/student/pace-feedback` with `requireStudentAuth` (cookie token session). The old route used `requireTeacherAuth` which returned 401 for students. Lesson: any component rendered on student pages must use student auth routes.
- **School calendar at teacher level, not school-admin (21 Mar 2026)** — Researched ManageBac (school-admin sets calendar, terms cascade to classes) and Toddle (similar school-level approach). StudioLoom's target user is a solo teacher, not a school admin. Simplified to per-teacher calendar setup in Teacher Settings. Quick templates (4 Terms, 2 Semesters, 3 Trimesters, Custom) make setup fast. Term assignment happens on the class-unit settings page, not the unit page — consistent with the unit-as-template architecture.
- **usePageData must use useMemo for stable references (21 Mar 2026)** — Bare object literals in React hooks create new references on every render, even when the underlying data hasn't changed. This triggers dependent useEffects (like usePageResponses) that reset state. The fix is `useMemo` + ref guards (`loadedPageRef` to only load once per page). This is a general React pattern: any hook returning an object that will be used in a dependency array must be memoized.
- **Anchor-based modular arithmetic for rotating cycles (21 Mar 2026)** — Cycle day = `((weekdaysBetween(anchor, target) % cycleLength) + anchorCycleDay - 1) % cycleLength + 1`. Only counts weekdays (Mon-Fri). Excludes dates in `excluded_dates` array. Optional per-term reset (re-anchors to Day 1 at start of each term). This handles 5/6/8/10-day cycles correctly across weekends, holidays, and term breaks.
- **iCal import for timetable, not LMS-specific integrations (21 Mar 2026)** — Rather than building ManageBac/PowerSchool/Toddle-specific API integrations, accept standard .ics files that any LMS can export. Lightweight RFC 5545 parser extracts VEVENT entries. Holiday detection via all-day events + keyword matching. Class matching via event SUMMARY → teacher's class names. Universal approach works for any school system.
- **Teacher settings: separate Workshop & Equipment tab (21 Mar 2026)** — Workshop spaces, tools & machines, and software sections moved from School tab to dedicated Workshop tab with wrench icon. Keeps school-level config (calendar, timetable) distinct from physical resources. Matt's request — workshop equipment is a separate concern from school structure.
- **Schedule overrides as JSONB on class_units, not a new table (22 Mar 2026)** — Lesson-level teacher adjustments (extra sessions for overtime, skip dates, notes) stored as `schedule_overrides` JSONB on the existing `class_units` junction table. Right granularity (per class-unit), avoids a new table + migration complexity. Shape: `{ extra_sessions: { pageId: N }, skip_dates: ["YYYY-MM-DD"], notes: { pageId: "text" } }`. Extra sessions default to 0 = 1 period per lesson; setting to 1 means "continues next class" (2 periods total).
- **Visual timetable grid over row-by-row entry (22 Mar 2026)** — TimetableGrid component presents Day 1-N columns × Period rows. Click empty cell to add class meeting (dropdown), click filled cell to remove. Much faster than the original approach of dropdowns per row. Color-coded by class (8-color palette). Quick-add row below grid for keyboard-first teachers. Both approaches (visual grid and quick-add) available simultaneously.
- **LessonSchedule uses server-side date computation (22 Mar 2026)** — Rather than duplicating cycle engine logic on the client, LessonSchedule calls the existing `/api/teacher/schedule/lessons?mode=next` endpoint to get the next N class dates. This ensures the cycle day calculation, excluded dates, and weekday-only logic stays in one place (cycle-engine.ts). The component requests enough dates to cover all sessions including extras from "continues" overrides.
- **Calendar templates always visible, not one-time setup (22 Mar 2026)** — SchoolCalendarSetup quick templates (4 Terms, 2 Semesters, 3 Trimesters, Custom) were hidden after first save, trapping teachers who deleted their terms. Fixed: templates always visible with dynamic label ("Quick Setup" → "Change Structure"). Teachers can change their term structure anytime without workarounds.
- **Copy-on-write forking for per-class content (22 Mar 2026)** — `class_units.content_data` defaults to NULL (inherits master). On first edit, `ensureForked()` deep-copies `units.content_data` into `class_units.content_data` with `forked_at` timestamp and `forked_from_version`. Same inheritance pattern as NM config (`class_units.nm_config` → `units.nm_config` → default). Resolution chain: `resolveClassUnitContent(masterContent, classContent)` returns `classContent ?? masterContent`. Pure function, no DB access.
- **Page ID preservation is a critical invariant during forking (22 Mar 2026)** — Progress records, NM checkpoints, due dates, schedule overrides, and student responses all reference page IDs. The deep copy in `ensureForked()` preserves all page IDs from master. Never regenerate or remap page IDs during fork. If master adds new pages later, they appear in master but not in the fork (fork is a snapshot).
- **Always use `.maybeSingle()` not `.single()` for optional rows (22 Mar 2026)** — Supabase `.single()` throws a PostgreSQL error when no row matches. `.maybeSingle()` returns null gracefully. This bit us on the class_units content query (unforked units have no content_data row match). General rule: use `.single()` only when you KNOW a row must exist (e.g., by ID after an insert). Use `.maybeSingle()` for any lookup that might legitimately return zero rows.
- **Dashboard 3-button layout: Teach / Manage / Edit Unit (22 Mar 2026)** — Replaced 5 buttons (Teach/Progress/Grade/Edit/Settings) with 3. "Manage" (class-colored solid fill) links to tabbed class-unit page (Overview/Progress/Grade/Safety tabs). "Edit Unit" (gray outline) links to class-local content editor. Reduces visual clutter on dashboard unit rows.
- **Tabbed Manage Class page over separate settings page (22 Mar 2026)** — Class-unit page at `/teacher/units/[unitId]/class/[classId]` now has tabs: Overview (NM config, schedule, settings), Progress (links to progress view), Grade (links to grading), Safety (badge management). Previously just a settings page. Tabs keep related functionality together without deep navigation.
- **Class-local editor as separate page, not inline (22 Mar 2026)** — Content editing for a class fork lives at `/teacher/units/[unitId]/class/[classId]/edit`, not embedded in the Manage page. Editing is a focused task that benefits from its own full-screen space. Shows fork status badge (amber "Class fork" vs blue "Using master template").
- **Content resolution wired everywhere class context exists (22 Mar 2026)** — Every route/component that reads `unit.content_data` in a class context must use `resolveClassUnitContent()`: class-unit settings page, Teaching Mode dashboard, projector view, student unit route, student units listing route. Missing any one breaks the fork illusion.

Attempting to handle both from a single client page causes "PKCE code verifier not found in storage" after full-page navigation chains (apex→www redirects, cookie scope boundaries). `AuthHashForwarder` sits on the landing route as a fallback for Supabase Site URL fallback cases, detects which payload type is present, and routes to the correct handler preserving both query and hash.
- **PKCE callback default next = set-password, not dashboard (15 Apr 2026)** — Supabase Site URL fallback strips the explicit `next=/teacher/set-password` and `type=recovery` params when `redirectTo` is not in the dashboard allowlist — only `?code=<uuid>` survives. Since PKCE is ONLY used in StudioLoom for the forgot-password flow, defaulting the callback's `safeNext` to `/teacher/set-password` is safer than `/teacher/dashboard`: a user who clicked a reset link ALWAYS needs to set a new password, regardless of whether the type hint survived. Route still honours explicit `type=recovery` and `type=invite`.
- **Public teacher paths allowlist (15 Apr 2026)** — `src/app/teacher/layout.tsx` now carries a `PUBLIC_TEACHER_PATHS` constant listing the four pages that must render without an authenticated session: login, welcome, forgot-password, set-password. `isPublicTeacherPath()` gates the three redirect/render checks. Without this, forgot-password flash-bounced to login before the form could submit because the teacher layout eagerly redirects anyone without a session. Same allowlist pattern used in the admin layout.
- **Change-password reuses set-password page (15 Apr 2026)** — Settings → General → Account → "Change password" links to `/teacher/set-password?next=/teacher/settings` rather than a separate change-password form. The set-password page already handles logged-in users setting a new password; the `next` param routes them back to settings after save. Avoids duplicate UI + validation logic. Only downside is the "Change password" framing reuses post-reset copy — acceptable tradeoff for ship velocity.
- **Preflight product name (20 Apr 2026, D-01)** — Originally picked "Bouncer" for stickiness + free verb infrastructure ("got bounced"). Same-day pivoted to **Preflight** because it's already a term of art in print/design (Adobe Acrobat Preflight, InDesign live preflight panels), pairs cleanly with the "Fabricator" role name via aviation metaphor, and avoids nightclub connotation. Naming cycle captured: Kiln → Platen → Tolerance → Sorter → Filter → Gate → Clearance → Greenlight → Bouncer → Preflight. Schema tables stay `fabrication_*` (domain-descriptive, grep-distinct); user-facing surfaces say Preflight.
- **Fabricator auth: own account type, not Supabase Auth (20 Apr 2026, D-05)** — Fabricators (lab techs) live in their own `fabricators` table with bcrypt `password_hash`. Cookie session pattern matches `student_sessions` (Lesson #4). Separation avoids forcing teachers to provision Supabase Auth for non-technical staff and keeps permission surface narrow (per-machine via `fabricator_machines` junction). Teachers invite via `invited_by_teacher_id` audit column; hard delete is service-role only (teachers can only deactivate via `is_active = false`). `/fab/login` own layout, not a teacher or student page.

Student-facing copy: *"Your scan results and preview thumbnails are kept permanently. Your original design file is deleted 30 days after the job is completed or rejected."*
