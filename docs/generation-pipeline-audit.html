The Core Contradiction

workshopPhases is REQUIRED in the schema — but lesson-structures.ts defines 8 different structures

The schema (buildPageContentSchema) always requires workshopPhases with exactly 4 fields: opening, miniLesson, workTime, debrief. These field names are hardcoded. Even if the AI prompt says "use Research structure: Mini-Lesson → Guided Investigation → Independent Analysis → Share", the schema forces the AI to output opening/miniLesson/workTime/debrief. The validation then maps these to the type-specific structure — but the AI never had the chance to generate type-specific phases.

Activity blocks have BOTH durationMinutes AND timeWeight — unclear which the AI should use

durationMinutes (number, optional) and timeWeight (quick/moderate/extended/flexible, optional) both exist on every activity section. The prompt says to use timeWeight as the primary signal, but durationMinutes is also described ("Estimated minutes for this activity"). The AI frequently outputs both. The validation only checks workshopPhases duration totals, not individual activity durations. Neither field is enforced or validated. The intent was: timeWeight replaces durationMinutes, but the schema still has both.

Skeleton Generation

The skeleton is the initial outline — lesson titles, types, and rough timing. Generated first, before individual lessons.

What the Skeleton Contains

  • title — lesson title
  • description — 1-sentence summary
  • designPhase — MYP phase (Inquiring, Developing, Creating, Evaluating)
  • lessonType — KEY FIELD workshop, research, ideation, skills-demo, making, testing, critique, presentation, assessment
  • suggestedActivities — activity outlines
  • estimatedDuration — "1 period", "2 periods"

Skeleton Constraints

  • max_tokens: 8192 → 16384 (auto-retry) HARDCODED
  • Activity count: ~1 per 10 min of usable time PROMPT
  • Model: always Sonnet 4 HARDCODED
  • lessonType inferred by AI from pedagogical context FLEXIBLE
↓ skeleton feeds into...

Journey Path ✅

Receives skeleton. Per-lesson prompt includes lessonType via getLessonTypeGuidance(). Validation uses lessonTypeMap.

Timeline Path ✅

Receives skeleton directly. lessonType flows to per-lesson prompt. But no post-gen validation.

Criterion Path ⚠

No skeleton used. No lessonType available. Always Workshop Model.

Pulse Repair Flow

When Pulse score is low, a repair prompt is injected into the NEXT lesson's generation.

Step 1: Score Lesson

Cognitive Rigour, Student Agency, Teacher Craft → each 0-10

Overall = mean − unevenness penalty

Step 2: Trigger Check

Overall < 5.0? → Repair

Any dimension < 4.0? → Repair that dimension

All good? → No repair needed

Step 3: Repair Prompt

Targets weakest dimension

Injects 2-3 Teaching Moves

"Replace or enhance one Work Time activity"

Repair only affects NEXT lesson, not current lesson

If the first lesson scores 3.0/10, the repair prompt is injected into lesson 2's generation. Lesson 1 ships as-is. This means the user sees a bad lesson if they don't regenerate it. The repair system is forward-looking only.

Key Findings & Recommended Actions

Finding 1: Schema forces Workshop Model phase names even for non-Workshop lessons

workshopPhases in buildPageContentSchema() always requires opening, miniLesson, workTime, debrief. When the prompt says "use Research structure: Mini-Lesson → Guided Investigation → Independent Analysis → Share", the AI still outputs under the Workshop field names. The validator then checks against type-specific rules but the AI never generated type-specific content — it generated Workshop content with the wrong names.

Fix: Either (a) make workshopPhases a dynamic schema keyed by lessonType (complex — Anthropic tool schemas are static), or (b) use generic phase names (phase1, phase2, phase3, phase4) with a phaseName field the AI fills in, or (c) switch to the Timeline schema for all paths (flat activities with role enum — already more flexible).

Finding 2: durationMinutes and timeWeight coexist with no resolution

Both fields exist on activity blocks. durationMinutes is a hard number ("15 minutes"), timeWeight is relative ("moderate"). The AI outputs both. Neither is validated. The intent was timeWeight replaces durationMinutes, but the migration never happened. The Timeline path actually REQUIRES durationMinutes (it's the only way to sum lesson time).

Fix: Pick one. If timeWeight is the future, remove durationMinutes from page/journey schemas and compute actual minutes from weight ratios during validation. If durationMinutes stays, remove timeWeight guidance from prompts. Don't have both.

Finding 3: Timeline path has ZERO post-generation quality checks

The Express/Timeline path (generate-timeline/route.ts) is the most popular path — it's what teachers get when they click Express mode. It generates flat activity lists with durationMinutes per activity. But it calls neither validateLessonTiming() nor computeLessonPulse(). Activities go straight to the client with no timing validation, no structure checking, no Pulse scoring, and no repair.

Fix: Add validation after timeline generation. This requires adapting validateLessonTiming() to work with flat activity arrays (currently expects workshopPhases), OR creating a new validateTimelineLesson() function.

Finding 4: Individual activity durations are never validated

The timing validation only checks phase-level totals (Workshop phases sum to usable time). Individual activities within Work Time can be any duration — a 45-minute lecture inside Work Time wouldn't be caught. The energy sequencing rules ("never two high-cognitive back-to-back") are prompt guidance only; nothing validates them.

Fix: Add per-activity validation rules: max duration per activity type, Bloom's level vs duration coherence (remember activities shouldn't be 30 min), energy sequencing check.

Finding 5: Bloom's taxonomy is correctly placed (on activities, not on time frames)

Good news: bloom_level IS on individual activity blocks as an optional enum. It's NOT on workshopPhases or time frames. The Pulse scoring reads bloom_level per activity and computes a weighted average for Cognitive Rigour. This matches your intent. The concern in your question ("I thought activity blocks were coded by Bloom's, not time frames") — this IS how it works. Activities have both bloom_level (cognitive demand) and optional durationMinutes (time). They're separate dimensions.

Finding 6: 60-min period length hardcoded in criterion path fallback

generate-unit/route.ts line ~115: const periodMinutes = 60; when no timing profile exists. The journey path uses 50 min. This inconsistency means criterion-generated lessons may be 10 min longer than journey-generated ones for the same class.

Finding 7: Criterion path cannot access skeleton lesson types

When a teacher uses the criterion-based generation path (rare — mainly for individual criterion regeneration), there's no skeleton and no lesson type. Every lesson validates as Workshop Model. If lesson-type-aware generation is the goal, this path needs a way to infer or accept lesson type.

Finding 8: The system architecture is sound but has accidental complexity

The 3-layer approach (prompt guidance → schema enforcement → post-validation) is correct. The problem is that the layers were built at different times and don't agree on structure. The schema says "Workshop Model always", the prompts say "use lesson-type structures", and the validation now checks lesson types. A clean rebuild would: (1) unify on a flexible schema (probably Timeline-style flat activities), (2) use lesson type to guide the AI, and (3) validate against the type-specific structure. The data fields (bloom_level, timeWeight, grouping, ai_rules, udl_checkpoints) are all correctly designed — the issue is only in the phase structure layer.
