### Migration discipline (v2 — timestamp prefixes)
- **New migrations use TIMESTAMP prefixes** (`YYYYMMDDHHMMSS_descriptor.sql`), not 3-digit numbers. Mint with `bash scripts/migrations/new-migration.sh <descriptor>`. Two parallel sessions cannot pick the same UTC second, which kills the collision class. Existing 000_..122_ migrations stay frozen — lex order means they sort BEFORE timestamps.
- **Claim immediately**: the moment `new-migration.sh` mints the pair, commit + push the empty stub to your feature branch BEFORE writing the SQL body. That reserves the timestamp on origin in seconds, not days. The script prints the exact commands.
- **Don't author 3-digit migrations** unless explicitly asked or doing emergency work — and if you do, run `bash scripts/migrations/next-free-number.sh` first AND check `/Users/matt/CWORK/.active-sessions.txt` for parallel sessions that might be racing for the same number.
- **Pre-merge gate**: before merging any feature branch into main, run `bash scripts/migrations/verify-no-collision.sh`. It exits non-zero on any same-prefix-different-filename collision against `origin/main`. Wire this into every Matt Checkpoint that includes migration work.
- **Don't run merges in the main worktree** (`/Users/matt/CWORK/questerra`) if you're mid-session and might get interrupted. An in-progress merge with conflicts can strand WIP for days. Either fast-forward via PR on origin, or use a throwaway worktree (`git worktree add ../questerra-merge main`) you delete after.
- **Migration applied-to-prod is the source of truth**, not the repo. Before renumbering or rolling back, verify what's actually on prod (Supabase dashboard, applied-migrations log).

### `sessionhandover` command
- Whenever the user types `sessionhandover` (or `session handover` / `handover` / `handoff`) on a line by itself, run this lightweight pickup ritual — no registry sync, no project-tracker update. Two steps:

  **Step 1 — Write the handoff file.** Resolve the filename: `docs/handoff/<current-branch>.md`, replacing `/` in branch names with `__` (so `feature/foo` → `docs/handoff/feature__foo.md`). Get the current branch via `git symbolic-ref --short HEAD` (fall back to `git rev-parse --short HEAD` + `DETACHED` if detached). OVERWRITE the file — it's pickup state, not history. Required content:
  - **Header**: `# Handoff — <branch>` followed by `**Last session ended:** <ISO timestamp UTC>`, `**Worktree:** <absolute path>`, `**HEAD:** <short-sha> "<commit subject>"`.
  - **What just happened** — 3–5 bullets describing the last meaningful changes from this session (commits, decisions, fixes, tests added).
  - **State of working tree** — `git status` summary (clean / staged / unstaged / mid-merge), current test count if known, pending-push count (`git rev-list --count @{u}..HEAD`), local-only branches if relevant.
  - **Next steps** — checkboxed todo (`- [ ] item`) for the next session, ordered. Include sub-bullets for context where it helps.
  - **Open questions / blockers** — anything waiting on a decision, a flaky external dependency, an unfinished investigation. If none, write `_None._`.

  **Step 2 — Emit the ready-to-paste pickup snippet.** After writing the file, output a single fenced code block (no prose around it, just the block) the user can copy verbatim into their next session. Format exactly:

  ````
  ```
  Read docs/handoff/<resolved-filename>.md and continue from where it left off.
  ```
  ````

  Where `<resolved-filename>` is the actual computed filename (e.g. `dashboard-v2-build.md`, not `<branch-name>.md`). One code block, nothing else after it. The user copies, pastes into the new session, and the new session picks up via the "Session pickup — READ FIRST" instruction at the top of this file.

## Loominary Context
StudioLoom is the first product in the **Loominary** product family (10 education apps sharing reusable code libraries). Each product is its own independently deployed app with its own database — "Loominary OS" is a shared package layer, not a runtime platform. Full Loominary docs live ONE LEVEL UP at `../Loominary/docs/`. **Important: always mount the parent folder (not just questerra/) so these `../Loominary/` paths resolve.** Key points for StudioLoom sessions:

- **Golden rule:** Build for StudioLoom with OS seams. Extract shared services only when product #2 (Makloom) forces it. Do NOT build abstract platform services. (ADR-001: `../Loominary/docs/adr/001-os-extraction-strategy.md`)
- **OS-seam principles to follow when coding:** (1) Keep domain-specific vocabulary configurable, not hardcoded. (2) Use `ServiceContext`-style patterns — tenant-aware, product-aware. (3) When building new services (auth, AI context, content delivery), keep interfaces clean enough that a second consumer could plug in without a rewrite. (4) Activity Blocks, FrameworkAdapter, and the content model are already good OS seams — maintain that pattern.
